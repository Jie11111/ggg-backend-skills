# Code Review Quality Checklist

本清单用于 `$ggg-code-review` 的代码质量补充检查。先完成 PRD/需求一致性、方案/任务一致性、diff 反查和幻觉审计，再按本清单检查本次 diff 的代码风险。

本文件不替代项目局部风格、`java-backend-code-standard` 和 `jzx-personal-java-style`。外部 review 原则只作为补充：优先发现正确性、数据一致性、安全、性能、兼容、可维护性和测试风险，不把个人偏好当成必须修。

## 1. 评审优先级

- 优先找会导致错误结果、线上风险、数据问题、安全问题、性能问题、兼容问题和不可测试的问题。
- 不把无关历史坏味道列成本次必须修；只在影响本次 diff 行为、回归或上线风险时记录。
- 没有文件和行号的问题，除全局验证缺口、文档缺口或发布风险外，不列为正式问题。
- 问题必须说明风险和具体改法，不能只写“建议优化”。

## 2. 功能和主链路

- 本次 diff 是否真正覆盖任务要求，边界条件、空值、状态流转、异常分支是否正确。
- Controller 是否薄，Service / Facade 主步骤是否清楚，复杂逻辑是否拆到语义明确的方法。
- 是否存在无效抽象、过度设计、滥用设计模式、只搬运参数的 Helper / Assembler / Util。
- 新增类是否职责清晰，是否把 Service、Manager、Helper 或 Util 做成万能类。
- 命名是否表达业务语义，布尔参数、魔法值、复杂条件和状态流转是否可读。

## 3. 接口、参数和兼容

- Controller 入参是否按项目约定校验，Service 是否补业务前置校验。
- `@RequestBody` DTO 字段类型、嵌套对象、数组、枚举、布尔值和 `Content-Type` 是否与接口文档一致。
- 前端禁止传的身份字段是否由后端推导；不能信任前端传入的 `userId/oid/classroomId/tenantId` 等关键身份。
- 新增字段、枚举值、默认值、空集合返回和错误码是否兼容旧调用方。
- 测试脚本是否安全发送 JSON，避免 Windows PowerShell 手拼 curl 导致字段名双引号丢失。

## 4. SQL、MyBatis 和数据

- 新 SQL 是否禁止 `SELECT *`，字段映射是否正确。
- 是否存在 N+1、循环查库、循环远程调用、循环写库或大结果集全量加载。
- Mapper 条件是否覆盖逻辑删除、租户/组织/用户隔离、状态过滤和权限范围。
- 批量查询是否空集合短路，分页是否有稳定排序。
- 新增字段、索引、默认值、历史数据处理是否与 `04-schema.sql` 和现有表结构一致。

## 5. 事务、异常和一致性

- 多表写入、状态联动、主记录和明细记录一起变更时，事务边界是否合理。
- 事务内是否新增长时间远程调用；必须调用时是否说明失败、重试、补偿和幂等边界。
- MQ、缓存、ES、WebSocket、事务后通知是否明确先落库还是先发事件，失败如何补偿。
- 业务异常和系统异常是否区分，系统异常日志是否带异常对象，是否吞掉堆栈。
- 局部降级或跳过外部依赖时，是否说明业务原因和影响范围。

## 6. 性能和资源

- 是否新增不必要远程调用，集合场景是否批量、缓存或合并请求。
- 是否在循环里查库、调 Dubbo/HTTP、访问 Redis/ES 或写高频日志。
- 是否打印完整 request/response/bizData 大对象。
- 大集合处理是否分页、分批或流式处理。
- 索引条件是否避免函数包裹、前缀模糊和类型隐式转换。

## 7. 安全和隐私

- 是否校验用户、组织、班级、课堂、课程、租户等归属，避免越权访问。
- 是否直接信任前端传入的身份、状态、金额、文件路径、URL、JSON 或枚举值。
- SQL、ES 查询、脚本、表达式、文件路径、URL 请求是否拼接用户输入。
- 日志、异常和返回值是否泄漏 token、Cookie、密码、手机号、身份证、学习隐私数据、内部配置或完整大对象。
- 重复点击、MQ 重投、定时任务重跑、事务后通知失败是否会重复写入或状态错乱。

## 8. 注释、日志和 Trace

- Controller、Service、Facade public 方法是否按项目习惯补必要中文方法注释。
- DTO、VO、实体、枚举、状态字段、配置项是否说明业务语义。
- 复杂状态流转、事务边界、异步/MQ、缓存、ES、WebSocket、分布式锁、幂等、旧链路隔离是否有边界和顺序说明。
- 注释是否解释业务边界和原因，而不是翻译代码；注释与代码冲突时列为问题。
- JZX 或已有 `TraceContextUtils` 的项目是否记录入口上下文、业务实例身份、状态变化、外部调用结果、异步事件 ID、批次/数量和最终结果。
- 主链路日志是否能定位“谁、哪个业务实例、什么状态、哪一步失败”。

## 9. 依赖、配置和发布风险

- `pom.xml`、依赖版本、scope、插件和构建配置是否必要，是否引入冲突、传递依赖或安全风险。
- Nacos/Apollo/本地配置、开关、灰度参数和默认值是否有发布说明和回滚方式。
- 新增配置是否有无配置兜底，是否改变旧链路默认行为。
- 新增定时任务、MQ topic、Redis key、ES index 是否有命名、生命周期、幂等和回滚说明。

## 10. 测试和验证证据

- 编译、单测、接口验证或关键链路验证是否覆盖本次高风险改动。
- 实现日志、测试报告或交付说明里写“通过/已验证”时，是否有命令、退出码、请求响应、日志、DB/ES/Redis 回查或截图证据。
- 缺少验证时，必须说明无法验证原因、风险和建议补跑命令。
