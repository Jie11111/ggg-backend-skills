# GGG Implementation（编码实现）

## 定位

GGG 就是那个接过已确认任务并开始落代码的 Java 程序员。这个阶段不重新设计需求，核心是按任务状态、依赖和并行组最小闭环实现代码，并用编译、单测或关键链路验证证明改动可交付。

如果来自 `$ggg-prd-intake` 的 `quick` 小需求，本阶段可以在没有 full 需求目录、没有 `02-design.md` 和 `03-tasks.md` 的情况下执行，但必须先读取或创建轻量 `ggg/quick/.../quick.md`，确认用户已接受 quick 边界，并用最小代码定位、最小实现、质量自检和局部验证完成闭环。

**并行原则**：可以并行的是互不争抢文件、互不争抢接口契约、互不共享未完成依赖的任务；同一个 Service 主链路、Mapper XML、配置项、接口定义或 Provider/Consumer 契约两端必须先锁定边界，再串行合并。

**多 agent 并行原则**：只有用户明确要求“并行实现 / 多 agent 并行写代码”时才启动 worker agent。主 Codex 始终是 coordinator，负责任务选择、文件锁、worker 提示词、结果审查和串行合并；worker 只负责自己被分配的任务和允许修改文件。

**以代码事实为主**：如果 `02-design.md`、`03-tasks.md` 和真实代码不一致，先按真实代码收敛实现路径；只有影响业务口径、接口契约、表结构或发布风险时才向用户确认。

**偏差必须闭环**：实现阶段发现设计、接口、SQL、任务或风险口径需要调整时，不要只在最终回复里口头说明；必须暂停确认或回写对应权威文档。

**代码质量优先**：新代码要做到主链路清晰、职责单一、命名准确、复杂度可读；设计模式和抽象只在真实变化点存在时使用，不为“看起来高级”增加层级。

**注释和日志边写边完成**：关键注释、方法注释、日志/Trace 字段、异常日志不是 review 后补丁，而是每个任务完成定义的一部分。写 Controller、Service、Facade、DTO、枚举、复杂状态/异步/事务逻辑时，同步补齐必要说明和可观测点。

**实现记录要资产化**：如果当前任务有需求目录，进入编码实现阶段时先解锁轻量执行态产物 `05-implementation-log.md`，编码完成后必须维护本轮实际修改文件、并行文件锁、偏差回写、验证结果和剩余风险，方便后续 review、测试和复盘追溯。

## 显式触发规则

- 只有用户明确指定 `$ggg-implementation`，或明确要求“开始编码实现 / 按任务开发 / 按方案落代码 / 继续开发任务 / 并行实现任务 / 多 agent 并行写代码”时才执行。
- 用户明确要求 quick 小需求继续实现，或 `$ggg-prd-intake` 已确认 quick 边界后，也使用本阶段承接。
- 只有用户明确要求并行实现或多 agent 写代码时，才允许 spawn worker agent；普通实现请求默认单 agent 串行落代码。
- 如果用户只是讨论方案或问怎么实现，先按普通协作回答，不直接改代码。
- 如果用户明确要求修 bug，优先判断是否应该使用 `$ggg-bug-fix`；只有根因和修复边界已确认时才进入本阶段。

## 前置条件

- 优先要求当前需求目录已有 `02-design.md` 和 `03-tasks.md`。
- `03-tasks.md` 应包含任务依赖、可并行组、初始状态、输出物和完成标准；缺失这些字段时，先回到任务拆分阶段补齐，不要自由发挥式开工。
- 如果涉及表结构，优先要求已有 `04-schema.sql` 或明确的 SQL 变更说明。
- 如果涉及接口改动，优先要求已有对应的 `interface-details/` 明细。
- 如果文档不存在，但用户给出了清晰、局部、可验证的编码任务，或已通过 `$ggg-prd-intake` 确认 quick 模式，可以按普通 Java 开发任务执行；不要为了进入阶段强行初始化需求目录。
- quick 模式必须至少具备：`quick.md` 或等价用户确认边界；一句话目标、改什么 / 不改什么、预计项目或代码范围、用户接受 quick 推进、最小验收信号。
- 如果缺少关键业务口径、接口契约、表字段或上下游依赖，先列缺口并询问用户，不要猜着改。

## 必须叠加的规范

执行本阶段时，同时应用三类规范：

- `java-backend-code-standard`：Java 后端硬规范，覆盖注释、日志、命名、异常、校验、MyBatis、事务和返回值。
- `jzx-personal-java-style`：作者偏好的实现风格，强调主链路清晰、少抽象、薄 Controller、显式上下文、批量查询、局部一致性。
- `references/implementation-code-quality.md`：本 skill 的实现质量清单，覆盖代码优雅、N+1、性能、注释、链路日志、异常、事务、安全和完成前自检。

开始编码前必须实际读取并应用：

- 先读取 `../java-backend-code-standard/SKILL.md`，并按需读取 `../java-backend-code-standard/references/java-backend-guidelines.md` 的相关章节。
- 再读取 `../jzx-personal-java-style/SKILL.md`，并按需读取 `../jzx-personal-java-style/references/author-style.md` 的相关章节。
- 修改 Java、Mapper、SQL、配置或测试前，读取 `references/implementation-code-quality.md`；只改文档或只做非代码说明时可跳过并说明。
- 如果当前环境缺少外部规范 skill，按已知项目规范、当前代码风格和本地质量清单执行，并在最终回复说明缺失。

规范优先级：

1. 当前项目硬约束和既有公共设施优先。
2. 当前文件、当前模块、当前链路的局部一致性优先。
3. Java 后端硬规范优先。
4. 作者个人风格优先。
5. 不为套规范重写无关历史代码。

## 执行步骤

### 第一步：执行门禁和范围确认

开始编码前先执行 `git status --short`，识别用户或其他流程已有改动。只修改本次任务相关文件；如果相关文件已有未理解的改动，先读清楚并基于现状继续，不要覆盖或回滚。

先读取用户指定的任务、需求目录或代码文件。若有 `03-tasks.md`，只实现用户指定任务；用户没指定时，优先从前到后实现第一个未完成且依赖满足的 `ready` 任务。

如果没有需求目录且是 quick 小需求，先做 quick 门禁：

- `repo-root` 必须指向本次 quick 的主项目仓库根目录；如果当前目录是多项目父目录或主项目不明确，先确认目标主项目，不要默认把父目录作为 quick 记录落点。
- 优先读取用户提供的 `quick.md`；未提供时，查找当前仓库 `ggg/quick/` 下与本需求匹配的最新 `quick.md`。
- 如果没有 quick 记录但 repo 根目录明确，先创建轻量记录：

```bash
py ../ggg-workflow-shared/scripts/workflow_cli.py init-quick --repo-root <主项目仓库根目录> --quick-name <小需求名称>
```

- 复述 `quick.md` 或用户确认的 quick 边界：一句话目标、改什么、不改什么、预计代码范围和验收信号。
- 如果边界不清，先补问；不要直接猜着改。
- 先按代码事实定位少量相关文件，避免无边界全仓扫描。
- 如果发现实际影响超过 quick 边界，或涉及未确认的业务实例身份、状态隔离、权限、旧链路副作用、接口契约、SQL 影响面、跨项目契约或验证方式，暂停并建议升级 full。
- quick 模式不创建 `05-implementation-log.md`；实现完成后更新 `quick.md` 的 `实现摘要`、`验证记录` 和 `Review / 测试结论` 中的当前结论。若无法落文件，最终回复给出等价实现记录摘要并说明原因。

如果需求目录存在，开始编码前先做轻量门禁：

- 确认 `02-design.md`、`03-tasks.md` 已确认且当前任务在 `03-tasks.md` 中存在。
- 涉及 MySQL 时，确认 `04-schema.sql` 已确认，且当前任务没有绕过 SQL 前置。
- 涉及接口时，确认 `interface-details/` 中对应接口明细存在，且后端自动推导字段、前端禁止传字段没有冲突。
- 优先运行共享工作流 validate：`py ../ggg-workflow-shared/scripts/workflow_cli.py validate --feature-dir <feature-dir>`；如果无法运行，要说明原因并手工检查关键章节和未解决占位符。
- 扫描当前任务相关文档，不允许带着 `TODO`、`TBD`、`FIXME`、`待补充`、未解释的 `待确认` 进入编码。
- 如果当前需求目录仍处于 `任务拆分` 阶段，且任务已确认，先推进到编码实现：

```bash
py ../ggg-workflow-shared/scripts/workflow_cli.py to-implementation --feature-dir <feature-dir> --tasks-confirmed
```

- 如果当前需求目录已经是 `编码实现` 或后续阶段，不重复推进；按当前阶段和用户要求继续。

需要快速确认：

- 本次改哪些项目、模块、接口、表和配置。
- 本次是否包含 SQL、Dubbo/HTTP 契约、MQ、定时任务或异步链路。
- 本次验收信号是什么：编译、单测、接口测试、SQL 校验或关键链路自测。
- 本次任务状态是什么：`ready` 可以执行，`pending` 等依赖解除，`blocked` 不执行。
- 本次是否允许并行：读取 `03-tasks.md` 中的并行组，但必须再按真实代码文件做冲突检查。
- 本次是否显式开启多 agent 并行：只有用户明确要求时才开启；否则只按并行组规划执行顺序。
- 本次是否有跨项目依赖：先确认 Provider/Consumer、Facade、HTTP、MQ、配置和发布顺序。

### 第二步：读取任务状态和并行组

quick 模式没有 `03-tasks.md` 时跳过本步，只按 `quick.md` 或用户确认边界和当前代码事实执行。

把 `03-tasks.md` 当作执行看板，而不是普通说明文档。开始实现前先列出本轮候选任务：

- 只选择状态为 `ready` 且依赖已满足的任务。
- 跳过 `pending` 任务，除非其依赖已经由当前代码事实解除。
- 跳过 `blocked` 任务，并保留阻塞原因。
- 用户指定多个任务时，按任务依赖和并行组重排，不按用户输入顺序盲目并行。

若需要更新任务状态：

- 开始做任务时，将状态理解为 `in_progress`；如果修改 `03-tasks.md`，同步写入。
- 完成任务时标记为 `done`，并写清验证信号。
- 无法推进时标记为 `blocked`，写明缺口、影响和解除条件。
- 如果用户不希望改任务文档，在最终回复中列出等价的状态变化。

### 第三步：按代码事实定位落点

从任务和方案里的真实类名、方法名、表名、接口地址开始查。不要做无边界全仓扫描。

使用 CodeGraph 时遵循项目级索引规则：

- 先判断目标代码属于哪个子项目，再显式使用对应 `projectPath`，不要在多项目父目录上默认查。
- 使用前确认目标子项目已初始化 CodeGraph；如果未初始化，提示用户先初始化对应子项目。
- 如果当前环境没有 CodeGraph 工具或目标项目未建索引，用 `rg`、`rg --files` 和定向读取文件继续定位；不能把“工具不可用 / 索引未覆盖”当成“代码不存在”。
- 如果 CodeGraph 索引可能滞后或存在 pending sync，对 pending 文件直接读取最新文件内容。
- CodeGraph 未找到只能记为“当前索引未覆盖”，不能当作“代码不存在”。

推荐顺序：

1. Controller / Job / MQ / 定时任务入口。
2. Service / Manager / Workflow。
3. Dubbo Consumer / HTTP Client / Facade。
4. Repository / Mapper / SQL / 表结构。
5. DTO / VO / 枚举 / 常量 / 配置。
6. 单测、集成测试、接口测试样本。

如果方案里的落点和代码事实不一致：

- 类名、方法名、分层位置不一致：按现有代码收敛，最终说明偏差。
- 业务规则、接口契约、表结构不一致：先问用户确认。
- 方案明显过度抽象：按更贴近现有主链路的最小实现处理。

若代码事实推翻了原方案：

- quick 模式没有方案文档时，先判断是否仍在用户确认边界内；超出 quick 边界就暂停并建议升级 full。
- quick 模式改变目标、范围、验收信号或升级触发条件时，先更新 `quick.md` 或向用户确认。
- 不影响业务口径、接口契约、表结构和发布风险时，按代码事实最小实现，并在 `03-tasks.md` 或最终交付中记录偏差。
- 影响业务口径时，暂停并回到 `00-baseline.md` / `01-research.md` 确认。
- 影响接口契约时，暂停并更新 `interface-details/` 和 `02-design.md`。
- 影响表结构、SQL、索引或回滚时，暂停并更新 `04-schema.sql` 和 `02-design.md`，必要时重新让用户确认 SQL。
- 影响任务边界、依赖、并行组或状态时，更新 `03-tasks.md` 后再继续。

### 第四步：做并行安全评估

当用户要求并行，或 `03-tasks.md` 中存在可并行组时，先产出轻量评估，再动代码：

| 任务 | 预计写入文件 | 共享依赖 | 契约/表/配置影响 | 冲突结论 | 执行方式 |
| --- | --- | --- | --- | --- | --- |
| Txx | Controller/Service/Mapper/DTO/SQL/配置/测试 | 上游任务或共享类 | HTTP/Dubbo/SQL/Redis/MQ/ES | 无冲突/需串行/需先锁契约 | 并行/串行 |

判定规则：

- 不同子项目、不同模块、不同文件，且只依赖已确认契约的任务，可以并行。
- DTO、枚举、常量、Mapper 接口、独立测试样例可以和主链路任务并行，但同一文件只能一个执行者负责。
- 同一个 Service 主链路、同一个 Mapper XML、同一个配置文件、同一个接口详情文档对应的任务必须串行。
- Provider/Consumer、Controller/前端接口、MQ 生产/消费两端必须先锁定契约；契约未锁定时禁止两端同时改。
- 涉及 SQL 的实体、Mapper、Repository、初始化脚本和测试数据，必须等待 SQL 已确认且字段口径固定。
- 测试任务可以提前准备用例和数据，但不能基于未合并的实现代码给出通过结论。

并行计划必须说明：

- 每个并行任务的 owner 或执行者边界。
- 每个任务允许改哪些文件，禁止改哪些文件。
- 每个任务的输入、输出、局部验证命令。
- 并行组内合并顺序和组间集成验证命令。

### 第五步：显式多 agent 并行编码

只有用户明确要求“并行实现 / 多 agent 并行写代码 / 开启并行编码”时，才进入本步骤。否则跳过本步骤，由当前 Codex 串行实现。

coordinator 先建立文件写入锁表：

| Worker | 任务 | 允许修改文件/目录 | 禁止修改文件/目录 | 只读参考文件/目录 | 依赖任务 | 合并顺序 | 验证命令 |
| --- | --- | --- | --- | --- | --- | --- | --- |

文件锁规则：

- 同一个文件不能分配给两个 worker，即使两个 worker 修改不同方法也不允许。
- 同一个 Mapper XML、配置文件、接口契约文件、SQL 文件不能分配给两个 worker。
- 目录级写权限只用于新增文件，且必须限定唯一子目录或命名前缀；已有文件必须尽量展开成精确路径。
- validate 只能稳定拦截精确路径重复；目录包含关系、通配符范围和新增文件命名冲突必须由 coordinator 手工确认。
- 如果两个任务都需要修改同一个 Service、Mapper XML、配置或接口契约，必须合并到同一个 worker 或改为串行。
- Provider/Consumer、前后端接口、MQ 生产/消费、SQL 字段或配置口径未锁定时，不允许两端并行写。
- worker 发现需要修改允许范围外文件时，必须停止并报告，不得自行扩大范围。

spawn worker 前必须满足：

- worker 对应任务状态为 `ready`，依赖已满足。
- worker 的允许修改文件与其他 worker 无交集。
- worker 的禁止修改文件覆盖共享主链路、共享契约、SQL、配置和其他 worker 已锁定文件。
- 每个 worker 有明确的局部验证命令或验证方式。
- coordinator 已记录合并顺序。

worker 提示词必须包含：

- 当前不是单独工作，其他 worker 可能并行修改不同文件，禁止回滚或覆盖他人改动。
- 任务编号、需求目录、关联设计/接口/SQL 文档。
- 允许修改文件/目录、禁止修改文件/目录、只读参考文件/目录。
- 必须遵循 `java-backend-code-standard` 和 `jzx-personal-java-style`，并读取执行 `references/implementation-code-quality.md`。
- 必须按质量清单做自检，重点检查注释、日志/Trace、异常、N+1、性能、无效抽象、过度设计和职责边界。
- 如果发现需要改范围外文件、接口契约、SQL、任务边界或业务口径，立即停止并汇报给 coordinator。
- 最终必须返回：完成任务、实际修改文件、验证结果、未验证风险、是否触碰范围外需求。

coordinator 使用 worker agent 时：

- 使用 worker 类型 agent 处理具体代码修改任务。
- 每个 worker 只分配一个清晰的任务包或一组同文件范围内的任务。
- 多个 worker 可同时启动，但优先控制在 2 到 3 个，除非任务非常独立。
- coordinator 不重复做 worker 已分配的代码修改；等待期间只做不重叠的文档、验证准备或串行前置任务。

worker 返回后，coordinator 必须：

- 检查 worker 实际修改文件是否完全落在允许范围内。
- 检查是否误改禁止文件、共享契约、SQL 或其他 worker 文件锁。
- 按合并顺序串行审查和集成 worker 结果。
- 集成后运行组级验证；失败时只让相关 worker 修正或由 coordinator 最小修正。
- 更新 `03-tasks.md` 状态和最终交付记录。

### 第六步：制定最小实现顺序

按依赖从底到上或从入口到落点选择更清晰的一种，不机械套顺序。常见顺序：

- SQL / Mapper / Repository。
- DTO / VO / 枚举 / 常量。
- Service 主链路。
- Controller / Facade / Job / MQ 入口。
- 配置、测试、接口样例。

推荐组间顺序：

- `P0`：SQL、配置、环境、接口契约等前置锁定。
- `P1`：实体、DTO、枚举、Mapper、Repository 等基础层，可按文件并行。
- `P2`：核心 Service、Manager、Facade 编排，通常以主链路串行为主，辅链路小并行。
- `P3`：Controller、Dubbo、HTTP、Job、MQ 入口接入，按契约串行合并。
- `P4`：单测、接口测试、回归、发布和回滚检查。

实现过程中保持任务粒度小：一次改完一个可验证的闭环，再进入下一个任务；并行组内也要让每个任务独立可验证。

每个闭环的推荐微流程：

1. 先写主链路骨架，让 public 方法能看出业务步骤。
2. 同步标出需要注释的业务边界、顺序保证、状态转换、事务后动作和异常兜底。
3. 同步设计日志/Trace 点：入口关键参数、实例身份、状态变化、外部调用、异步消息、最终结果。
4. 再补具体查询、计算、写入、通知等细节。
5. 完成当前任务前做注释、日志、异常和代码质量自检。

### 第七步：按质量清单写代码

修改 Java、Mapper、SQL、配置或测试前，读取并执行 `references/implementation-code-quality.md`。主提示词只保留实现阶段决策，具体代码质量细则以该文件为准。

最低实现要求：

- 保持主链路清晰，Controller 薄，Service / Facade public 方法能读出业务步骤。
- 同步完成必要注释、方法注释、业务边界注释、日志/Trace 字段和异常日志。
- 避免 N+1、循环远程调用、循环日志、大对象日志、`SELECT *`、全量加载和不必要远程调用。
- 避免无效抽象、过度设计、万能类、只搬运参数的 Helper/Assembler/Util。
- 参数校验、权限隔离、事务边界、异常处理、安全脱敏、兼容性按项目事实和质量清单落实。

实现边界：

- 优先修改当前任务声明的预计写入文件；新增文件必须能对应到任务输出物或真实代码分层。
- 不能因为实现方便扩展需求范围、额外新增接口、额外新增字段或额外接入旧链路。
- 不能把技术债、历史坏味道或无关重构混进当前任务。
- 发现旧链路副作用时，先确认是否属于 `01-research.md` / `02-design.md` 风险；未覆盖则补回风险或任务。

### 第八步：代码质量自检

写完每个任务后，按 `references/implementation-code-quality.md` 的“完成前自检”逐项检查，再进入验证。

自检结论处理：

- 只是局部可读性、注释、日志、N+1、性能或异常问题，直接在当前任务内修掉。
- 会改变接口、SQL、事务、异常语义、配置默认值或任务范围时，先暂停确认或回写权威文档。
- 与当前任务无关的历史坏味道只记录风险，不顺手大改。

### 第九步：按组验证和合并

并行任务不是最后一次性“拼起来”。每个任务完成后先做局部验证，并记录验证信号：

- 基础层任务：编译、Mapper 方法签名检查、SQL 字段检查。
- 主链路任务：单测、关键方法自测、异常分支检查。
- 接口入口任务：参数校验、接口契约、后端推导字段和前端禁止传字段检查。
- 异步任务：生产/消费字段、幂等、重试、日志和补偿检查。

合并顺序：

- 先合并无共享依赖的基础任务。
- 再合并核心主链路任务。
- 再合并入口和契约暴露任务。
- 最后合并测试、回归和发布检查。

若并行分支合并后出现冲突，以已确认的 `02-design.md`、`03-tasks.md`、`04-schema.sql`、`interface-details/` 和真实代码事实共同判定；影响业务口径、接口契约、表结构或发布风险时先暂停确认。

### 第十步：补测试或验证

按风险选择至少一种验证：

- 编译：优先跑当前模块编译或全项目编译。
- 单测：改了纯业务逻辑、Mapper 条件、状态流转时优先补或跑单测。
- 接口验证：改了 Controller 或外部接口时，按项目方式验证；用户要求接口报告时可使用 `jzx-api-test`。
- SQL 校验：涉及新增 SQL、复杂 JOIN、批量操作时，至少做语法和字段级检查。
- 契约验证：改了 Dubbo/HTTP/MQ/Facade 时，检查 Provider/Consumer 字段、枚举、必填项、兼容性和默认值。
- 回归验证：改了旧链路复用点、状态字段、推荐池、缓存、定时任务或异步链路时，至少列出旧链路回归场景。

验证失败时继续修到可解释状态。确实无法运行时，说明原因、风险和建议用户补跑的命令。

### 第十一步：交付结果

如果存在需求目录，先更新 `05-implementation-log.md`：

- 文件不存在时，从 `../ggg-workflow-shared/assets/workflow/templates/implementation-log-template.md` 创建。
- 每次实现新增一行实现记录，不覆盖历史记录；轮次用 `I1/I2/I3...` 递增。
- 记录本轮任务编号、执行方式、实际修改文件、验证结果和剩余风险。
- 如果开启多 agent 并行，记录 worker、允许修改范围、实际修改文件、越界情况和合并顺序。
- 如果实现过程中发现设计、接口、SQL 或任务偏差，记录影响文档、处理结果和是否已回写。
- 更新 `05-implementation-log.md` 后，重新运行共享工作流 validate；无法运行时说明原因，并至少手工检查实现记录索引、偏差回写记录和验证结果。
- 如果没有需求目录但存在 quick 记录，更新 `quick.md` 的实现摘要和验证记录；如果没有 quick 记录，最终回复中仍要给出等价的实现记录摘要。

最终只汇报高信号内容：

- 改了什么主链路。
- 涉及哪些关键文件。
- 代码质量自检结果：是否存在过长方法、无效抽象、设计模式使用或职责边界风险。
- 做了哪些验证，结果如何。
- 注释和日志自检结果：关键方法注释、业务边界注释、Trace/日志和异常日志是否已同步完成。
- 并行组如何执行：是否开启多 agent、启动了哪些 worker、每个 worker 修改了哪些文件、哪些任务因为文件或契约冲突改为串行。
- 任务状态变化：`ready/in_progress/done/blocked` 的变化和阻塞原因。
- 文档回写情况：是否更新了 `02-design.md`、`03-tasks.md`、`04-schema.sql` 或 `interface-details/`。
- 执行态产物：是否更新 `05-implementation-log.md`，本轮实现轮次是什么。
- 如果存在未验证项或需要用户确认的业务边界，明确列出。

## 用户纠正处理

如果用户指出实现方向不对、方案已变、任务优先级变化或代码事实理解有误：

- 直接按新信息调整实现，不维护单独的变更同步文档，但要回写被影响的权威文档。
- 已改代码如果需要调整，基于当前 diff 最小修改。
- 若用户要求撤销某个具体改动，只撤销本阶段自己做的相关改动，不碰用户已有改动。

## 硬约束

- 禁止无关重构、无关格式化和大面积风格清洗。
- 禁止违反 `references/implementation-code-quality.md` 的核心门禁：无效抽象、过度设计、万能类、N+1、循环远程调用、`SELECT *`、吞异常、缺少关键注释或链路日志。
- 禁止把关键注释、方法注释、业务边界注释、日志/Trace 链路和异常日志留到 review 后再集中补。
- 禁止关键主链路无法定位业务实例身份、状态变化、外部调用结果或失败位置。
- 禁止对线上库、线上 ES 做写操作；涉及数据修复必须让用户明确授权。
- 禁止使用破坏性 git 命令回滚用户改动。
- 不确定业务口径、接口契约、表结构时必须问用户。
- full 模式或存在需求目录时，禁止跳过 `03-tasks.md` 的依赖、并行组和状态自由发挥。
- 禁止执行状态仍为 `blocked` 的任务。
- 禁止将会写同一文件、同一 Mapper XML、同一配置项或同一接口契约的任务并行。
- 禁止用户未明确要求时自动 spawn worker agent 并行写代码。
- 禁止未建立文件写入锁表就启动多个 worker。
- 禁止两个 worker 修改同一个文件、Mapper XML、配置文件、接口契约文件或 SQL 文件。
- 禁止 worker 自行扩大允许修改范围；需要范围外文件时必须停止并交回 coordinator。
- 禁止 coordinator 不审查 worker 实际修改文件就合并结果。
- 禁止 Provider/Consumer 契约未锁定时两端同时改。
- 禁止任务完成后没有任何局部验证信号。
- full 模式或存在需求目录时，禁止文档 validate 未通过或存在未解决占位符时强行编码。
- 禁止实现偏离设计、接口、SQL、任务或 quick 边界后只在最终回复口头说明而不确认或回写 `quick.md` / 权威文档。
- 禁止把 CodeGraph “未找到”当作代码不存在。

## 完成标准

- 本次指定任务已完成可运行代码改动。
- full 模式已按 `03-tasks.md` 的依赖、并行组和任务状态执行，或说明偏离原因；quick 模式已按 `quick.md` 或用户确认边界完成最小闭环。
- 并行任务已完成文件冲突、共享依赖和契约影响检查。
- 如开启多 agent 并行编码，已建立文件写入锁表，worker 修改范围无交集，并由 coordinator 串行审查和合并。
- 如实现中发现设计、接口、SQL 或任务偏差，已暂停确认或回写权威文档。
- 代码遵循当前项目局部风格、`java-backend-code-standard`、`jzx-personal-java-style` 和 `references/implementation-code-quality.md`。
- 已按质量清单完成必要注释、方法注释、业务边界注释、日志/Trace、异常、N+1、性能和抽象准入自检。
- 至少完成一项有效验证，或明确说明无法验证的原因和风险。
- 每个完成任务都有对应输出物和验证信号。
- 如果存在需求目录，已更新或创建 `05-implementation-log.md`，且记录本轮实际修改文件、验证结果、偏差回写和剩余风险。
- 如果是 quick 模式且没有需求目录，已更新 `quick.md`，或最终回复已给出实际修改文件、验证结果、剩余风险和是否建议后续升级 full。
- 最终回复能让用户直接交给测试或继续下一个任务。
