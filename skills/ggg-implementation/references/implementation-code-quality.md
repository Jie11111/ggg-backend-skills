# Implementation Code Quality Checklist

本文件用于 `$ggg-implementation` 编码阶段。开始修改 Java 后端代码、Mapper、SQL、配置、测试前读取；只改文档或只做非代码说明时可以跳过，并在最终回复说明。

本清单是 `$ggg-implementation` 的实现阶段补充门禁，不替代项目局部风格、`java-backend-code-standard` 和 `jzx-personal-java-style`。外部规范由主流程负责加载，本文件只保留编码时最容易漏掉的质量检查项。

## 1. 主链路和代码风格

- Controller 保持薄，只做入参接收、边界校验、上下文提取、调用 Service、包装返回。
- Service / Facade public 方法先呈现业务步骤，让主流程能读出“校验、查询、计算、写入、通知、返回”。
- 复杂细节下沉到语义明确的私有方法；方法名表达业务动作，不使用 `handleData`、`processInfo`、`buildResult` 这类空泛名字。
- 优先用 guard clause 降低嵌套；嵌套超过 3 层、一个方法混合查询/计算/写入/通知时，优先拆分。
- 上下文参数优先显式传递，例如 `userId`、`oid`、`employeeId`、`classroomId`，不要在深层方法反复隐式获取。
- 命名体现业务语义，避免孤立的 `data`、`list`、`map`、`flag`、`type`。
- 返回值和异常边界清楚；不要用 `null` 表示多种业务状态。

## 2. 代码优雅和抽象准入

- 只有存在明确变化点、多个实现、可替换策略或项目已有同类模式时，才引入策略、模板、工厂、接口或抽象类。
- 单实现、一次性逻辑、短期没有第二个分支时，不新增 Strategy、Factory、Manager、Helper、Util。
- 不为了消灭一个 `if` 引入多层模式；简单分支直接写清楚更好。
- 不新增只搬运参数、只改名、不承载业务语义的私有方法、Assembler、Converter、Helper。
- 复用代码前先确认业务语义一致；只是代码形状相似但业务含义不同，不强抽公共方法。
- 新增类必须职责清晰，不把 Service、Manager、Helper 或 Util 做成万能类。
- 魔法值优先改成枚举、常量或配置；只出现一次且语义明确的局部值不强行抽常量。

## 3. SQL、MyBatis 和 N+1

- 新 SQL 禁止 `SELECT *`，必须显式字段。
- 避免循环查库、循环远程调用和循环写库；集合场景优先批量查询、批量写入、JOIN、范围查或一次拿齐后内存映射。
- Mapper 条件必须覆盖逻辑删除、租户/组织/用户隔离、状态过滤和权限范围；旧链路复用时特别检查默认条件是否会误伤。
- 批量查询要注意空集合短路，避免生成非法 SQL。
- 分页接口必须有稳定排序；大结果集不能一次性全量加载。
- 索引相关字段要和 `04-schema.sql` / 现有表结构一致，避免在索引列上做不必要函数、模糊前缀破坏或类型隐式转换。
- 新增字段、枚举值、默认值、逻辑删除字段、时间字段要和目标模块已有风格一致。
- 涉及复杂 JOIN、动态 SQL、批量更新时，至少做字段级和语法级自查，能跑测试就跑测试。

## 4. 性能和资源

- 不在主链路新增不必要远程调用；必须调用时先批量、缓存或合并请求。
- 不在循环中写高频日志、查库、调 Dubbo/HTTP、访问 Redis/ES。
- 避免记录完整 request/response/bizData 大对象；只记录业务实例身份、状态、数量和关键结果。
- 缓存、Redis、ES、MQ 只在方案已确认时接入；必须写清 key、过期、幂等、失败处理和一致性边界。
- 异步任务、MQ、定时任务要考虑重复执行、重试、乱序、幂等和补偿。
- 大集合处理要关注内存；必要时分页、分批或流式处理。

## 5. 参数校验、安全和兼容

- Controller 入参按项目约定使用 `@Valid` 或等价校验；Service 层补业务前置校验。
- 前端禁止传的身份字段必须由后端推导；不要信任前端传入的 `userId/oid/classroomId/tenantId` 等关键身份。
- 权限和归属校验要覆盖用户、组织、班级、课堂、课程、租户等真实隔离维度。
- 枚举、状态、金额、数量、时间范围、文件路径、URL、JSON 字段必须校验合法性。
- 日志、异常、返回值不能泄露 token、Cookie、密码、手机号、身份证、学习隐私数据、内部配置或大对象。
- 接口新增字段、枚举值、默认值和空集合返回要兼容旧调用方。

## 6. 事务、异常和一致性

- 多表写入、状态联动、主记录和明细记录一起变更时，使用项目约定事务；Spring 项目优先 `@Transactional(rollbackFor = Exception.class)`。
- 事务里避免长时间远程调用；必须调用外部服务时，说明失败、重试、补偿和幂等边界。
- 业务异常和系统异常区分处理；系统异常不能吞，日志必须带异常对象。
- 局部降级、跳过外部依赖、吞掉可接受异常时，必须用注释或日志说明业务原因和影响范围。
- MQ、缓存、ES、WebSocket、事务后通知要明确顺序：先落库还是先发事件，失败如何补偿。

## 7. 注释

- Controller、Service、Facade public 方法按项目习惯补必要中文方法注释，说明业务语义、参数、返回值和关键异常。
- DTO、VO、实体、枚举、状态字段、配置项要说明业务语义；状态值和类型值不能只写数字。
- 复杂状态流转、事务边界、异步/MQ、缓存、ES、WebSocket、分布式锁、幂等、旧链路隔离必须写业务边界和顺序保证。
- 注释解释“为什么这样做、边界是什么、顺序为什么不能变、兜底何时触发”，不翻译语法。
- 不写“查询数据”“设置变量”“调用方法”这类废话注释。
- TODO/FIXME 必须带责任人和日期；能当场收口的不要留下。

## 8. 链路日志和 Trace

- 开始编码前识别项目日志体系：`groupId` 包含 `jzx` 或已有 `TraceContextUtils` 时，业务链路优先写 trace 字段；其他项目沿用现有 `log.info/warn/error` 风格。
- 入口处记录关键上下文：`userId/oid/employeeId/classroomId`、业务实例 ID、请求来源、关键参数摘要。
- 关键节点记录状态变化、核心计算结果、外部调用结果、MQ/Redis/ES/WebSocket 事件 ID、批次号、数量和是否命中兜底。
- 结束处记录关键结果：新建记录 ID、最终状态、数量、异步事件是否触发。
- 业务异常用 `warn`，系统异常用 `error`，必须带异常对象；不能只打印 message。
- 禁止循环高频日志，禁止打印敏感信息和完整大对象。
- 主链路必须能定位“谁、哪个业务实例、什么状态、哪一步失败”。

## 9. 完成前自检

完成每个任务前至少自查：

- 主链路是否清楚，Controller 是否薄，Service 是否能读出业务步骤。
- 是否存在 N+1、循环远程调用、循环日志、大对象日志或全量加载。
- 新 SQL 是否显式字段，是否带必要隔离条件和逻辑删除条件。
- 注释是否覆盖 public 方法、DTO/枚举、复杂边界和顺序保证。
- Trace/日志是否覆盖入口、关键状态变化、外部调用和最终结果。
- 异常是否带堆栈，是否区分业务异常和系统异常。
- 是否引入无效抽象、过度设计、万能类或只搬运参数的工具层。
- 是否改变接口、SQL、配置、事务或任务范围；如改变，先暂停确认或回写权威文档。
