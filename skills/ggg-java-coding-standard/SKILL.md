# GGG Java Coding Standard

## 定位

把本文件作为 GGG 唯一的 Java 后端代码规范。它提供共享规则，不创建流程阶段、不产生独立文档、不要求用户在 implementation 之外再次确认。

目标是写出正确、直接、主链路清楚且便于验证的代码。不要把个人偏好包装成普遍真理，也不要为了形式统一破坏项目现有约定。

## 规则优先级

按以下顺序处理冲突：

1. 已确认的需求、接口契约、SQL 和技术决策。
2. 编译器、框架、数据库及运行环境的真实约束。
3. 仓库 `AGENTS.md`、项目构建、格式化、静态检查和测试配置。
4. 当前模块、相邻链路和同类代码的稳定约定。
5. 本规范。

局部历史写法不能成为继续引入正确性、安全、性能或兼容性问题的依据。遇到历史坏味道时只修本次改动所需的最小范围，不顺手清洗整个文件。

本规范只有两类规则：

- **硬约束**：涉及需求正确性、接口兼容、数据一致性、权限安全、敏感信息和明显性能风险，必须满足；无法满足时暂停并说明。
- **上下文约定**：命名、注入方式、注释语言、日志设施、返回包装和分层形态，跟随项目事实；不要凭公司样本猜测。

## 编码前先建立事实

开始修改前：

1. 读取当前任务或 quick 边界、完成标准、接口明细和 SQL。
2. 定向读取目标类、直接调用方、被调用方、DTO、Mapper、配置和相邻测试。
3. 读取项目已有 formatter、Checkstyle、PMD、SpotBugs、测试和构建命令。
4. 确认当前模块采用的注入、异常、日志、返回体、持久化和注释语言。
5. 写清本次必须改变的行为和明确不改变的行为。

代码事实与已确认设计冲突时，不要自行重新解释业务：仅代码落点变化可按事实最小调整；影响业务规则、接口、SQL、事务或任务范围时回到上游确认。

## Java 后端编码前实现预检补充

实现预检由 `ggg-implementation` 按 `tiny / normal / high` 自适应执行。涉及 Java 后端时，只把本次 diff 真正触发的专项判断写进同一份预检，不新增表格、文档或确认步骤：

- 主链路或职责变化：明确校验、查询、计算、写入、通知和返回如何衔接，并核对 Controller、Service、Facade、Mapper 等真实职责。
- 数据、契约或失败边界变化：检查事务、幂等、并发、异常、兼容调用方及外部调用边界。
- `high` 风险触发性能或恢复面：核对 DB、RPC、缓存次数、循环 I/O、批量、分页、超时、补偿和恢复方式。
- `high` 风险触发抽象或方案偏差：说明直接实现或复用现有模式的事实依据，以及与已确认方案是否一致。
- 每个档位都写清与当前行为相称的验证策略。

`tiny` 不补填 normal/high 的不适用清单；局部低风险改动完成两面预检后直接继续。不要把实现草图变成第二份技术方案。

设计判断必须引用当前代码、已确认方案或项目约定。数据规模、流量、索引效果和运行行为未知时明确写未知，不编造性能结论。只有同时存在真实变化点、多个实现或项目已有稳定范式时才采用策略、工厂、模板、责任链、状态等模式；说明变化点和更简单方案为什么不足。只有一个简单分支时直接实现。

只有相关事项尚未在已确认需求、技术方案、SQL、接口或任务中明确，或者实际代码事实迫使实现偏离已确认内容时，才暂停编码并请用户确认。已经确认且代码事实一致的内容直接实施，不重复询问。

- 业务语义、实例身份、权限或状态规则仍不明确。
- 公共接口、表结构或索引、事务一致性、缓存、MQ、异步顺序或安全边界缺少已确认决策，或者实现需要偏离已确认决策。
- 跨模块共享抽象、架构替换、新设计模式或新基础设施没有得到上游确认。
- 性能方案依赖尚未查证的数据量、调用频率、热点分布或运行配置。

## 总体代码设计

### 主链路优先

- 让入口方法能直接读出核心业务步骤，例如校验、查询、计算、写入、通知和返回。
- 使用 guard clause 处理非法条件和提前返回，避免无意义的深层嵌套。
- 把复杂细节下沉到名称表达业务动作的私有方法；避免 `handleData`、`processInfo`、`buildResult` 等空泛名称。
- 显式传递必要的业务上下文和实例身份；不要在深层方法反复隐式获取用户、组织、课堂或租户信息。
- 不用 `null` 同时表达不存在、失败、未初始化等多个业务状态；沿用项目已有的明确返回或异常语义。

### 最小设计

- 只实现当前任务需要的闭环，不为假设中的未来需求预留多层扩展。
- 只有存在真实变化点、多个实现、替换需求或项目已有同类模式时，才引入策略、工厂、模板、接口或抽象类。
- 不创建只换名字、搬参数或转调一行代码的 Helper、Manager、Assembler、Converter、Util。
- 代码形状相似不等于业务语义相同；复用前核对数据来源、过滤范围、异常、顺序和兼容性。
- 一个清晰的简单分支优于为消灭 `if` 引入多层模式。
- 同一业务语义重复出现时优先复用已有枚举或常量；只出现一次且含义清楚的局部值不强制抽常量。

### 纵向闭环

按任务逐个形成可运行闭环，不跨全部任务机械地先写完所有 Mapper、再写所有 Service、最后写所有 Controller。

单个任务内部根据真实依赖完成：

1. 明确接口或调用契约。
2. 实现核心业务规则和状态变化。
3. 补齐数据访问与外部依赖。
4. 接通入口和返回结果。
5. 按变更风险补齐必要测试，并使用编译、静态检查、接口或运行验证完成局部闭环。

## 分层职责

### Controller 或入口

- 只处理接口边界：接收入参、结构校验、提取可信上下文、调用业务能力、按项目约定返回。
- 不承载业务状态机、持久化组合、跨服务编排或复杂对象转换。
- 不信任前端传入的用户、组织、课堂、租户等身份字段；按已确认契约从服务端上下文推导。
- 简单边界类型转换可以沿用项目已有适配方式；涉及业务校验、错误码或多字段组装时下沉到 Service、Facade 或已有适配层。

### Service、Workflow 或领域能力

- public 方法表达完整业务语义和主要步骤。
- 把业务校验、状态转换、幂等、写入顺序和外部调用边界集中在可读主链路中。
- 私有方法按业务动作命名，并保持输入、输出和副作用清楚。
- 不为了单个外部调用无限扩张内部 Service API；需要边界适配时使用项目已有 Facade 模式。

### Facade、Client 或 Provider

- 保持契约窄而稳定，负责跨项目 DTO、错误语义和类型边界。
- 不向调用方泄露内部实体、Mapper 对象或实现细节。
- Provider 与 Consumer 同时核对方法签名、序列化字段、枚举值、空值和版本兼容。
- 项目没有 Dubbo、Feign 或同类设施时，不生造对应注解和基础层。

### Repository、Mapper 和数据访问

- 方法名体现主要过滤条件和业务含义，不使用含糊的 `queryData`、`getList`。
- 返回值遵守项目约定；集合查询通常返回空集合，不用 `null` 表达“没有数据”。
- 保持数据访问职责明确，不把业务状态机塞入 Mapper，也不为了分层形式制造无意义转发层。

## 接口、DTO 和兼容性

- 以已确认接口契约为准，不根据字段名机械更改类型。
- 前端或跨语言边界中的大整数 ID 优先使用能避免精度丢失的契约类型；已发布接口保持兼容，内部 Entity 和 Mapper 使用数据库真实类型。
- 明确必填、可空、默认值、枚举、状态、数量、时间范围和空集合语义。
- 新增字段或枚举值时检查旧调用方反序列化、默认分支和未知值处理。
- DTO、VO、Request、Response 不直接暴露敏感字段或内部配置。
- JavaBean、Jackson、Dubbo 等存在命名或序列化风险的字段，核对真实注解和序列化结果，不凭方法名猜测。

## 参数、权限和安全

- 在接口边界做结构、格式和范围校验，在业务层做状态、归属和业务前置校验。
- 权限检查覆盖真实隔离维度，例如用户、组织、班级、课堂、课程和租户。
- 对不可信输入使用允许列表、长度和范围限制；SQL 使用参数绑定，不拼接用户输入。
- 文件路径、URL、JSON、表达式和模板输入必须检查合法性及访问边界。
- 日志、异常和返回值不得泄露 token、Cookie、密码、手机号、身份证、学习隐私、密钥和内部配置。
- 不用前端隐藏按钮代替后端权限校验。

## SQL、MyBatis 和性能

- 新 SQL 显式列出字段，不写 `SELECT *`。
- 带上已确认的逻辑删除、租户、组织、用户、状态和权限过滤条件。
- 避免循环查库、循环写库、循环 RPC/HTTP、循环 Redis/ES 和循环日志；优先批量查询、批量写入、JOIN、范围查询或一次获取后内存映射。
- 批量查询前处理空集合，避免非法 `IN ()`；批量大小遵守项目和数据库限制。
- 分页或分批处理使用稳定排序；大结果集不要一次性全量加载。
- 核对字段类型，避免索引列函数、隐式类型转换和无依据的前缀模糊查询破坏索引。
- 只有掌握真实查询条件、数据规模和现有索引时才新增或调整索引；不要凭感觉设计 SQL 优化。
- 新增字段、默认值、枚举和时间语义必须与 `04-schema.sql` 及真实表结构一致。

### DDL 和公共字段

新增或修改表结构时，先记录同模块参考表及字段风格，不得只看 Java 实体或凭个人习惯命名。目标模块没有可参考表时，使用以下兜底公共字段：

```text
id、create_user、create_time、modify_user、modify_time、is_delete
```

- 参考表使用其他命名时以项目事实为准，并在实现记录中写明参考表和差异。
- 系统流水、Outbox、幂等、临时或追加写表确实不需要逻辑删除或操作人字段时可以省略，但必须逐表说明生命周期和省略原因；不能静默缺失。
- MySQL 交付 DDL 必须核对字段与表注释、枚举取值说明、主键/唯一键/索引、`InnoDB` 和 `utf8mb4`；测试初始化脚本必须标明测试方言，不能冒充生产 MySQL DDL。
- 同时存在 `schema.sql`、迁移脚本、Entity、Mapper 或 Repository 时，逐字段核对名称、类型、默认值、可空性和时间更新语义。
- quick 没有已确认 SQL、项目参考表或可靠兜底依据时暂停确认；不能因为能在 H2 或单测中建表就认定 DDL 合格。

## 事务、并发和异步一致性

- 根据真实写入边界使用项目已有事务约定，不机械给所有方法增加事务。
- 多表写入、主从记录或状态联动需要原子性时，明确事务范围和回滚语义。
- 避免在长事务中执行慢远程调用；无法避免时说明失败、重试、补偿和幂等边界。
- 状态更新检查重复提交、并发覆盖、条件更新和影响行数。
- MQ、Job、异步任务和回调考虑重复执行、重试、乱序、幂等、死信或补偿。
- 明确先落库还是先发事件、事务提交前还是提交后通知；在非显然代码处解释顺序原因。
- Redis、ES、缓存和数据库同时参与时，按已确认设计实现一致性，不自行增加“双写兜底”。

## 异常、日志和 Trace

- 沿用项目已有业务异常、错误码和全局异常处理；项目没有对应基础设施时不要生造公司同名类。
- 不吞系统异常，不用空 catch 掩盖失败；有意降级时明确允许失败的业务原因、影响和可观测信号。
- 避免在多层重复记录同一异常；在能提供业务上下文且不会重复的位置记录。
- 系统异常日志保留异常对象；业务异常是否记录以及使用 `warn` 或其他级别跟随项目约定。
- 记录业务实例 ID、状态、数量、批次和关键外部结果，不打印完整 request、response 或大对象。
- 禁止循环高频日志和敏感信息日志。
- 只有项目真实存在并稳定使用 `TraceContextUtils`、`@CustomLog` 等设施时才启用；不要仅凭 `groupId` 或仓库名称推断。
- 主链路应能定位“哪个业务实例、处于什么状态、在哪一步失败”，但不要为每个方法机械增加入口和出口日志。

## 注释和 Javadoc

- 对外接口、公共契约、复杂业务方法和容易误解的 DTO/VO 字段写有业务价值的 Javadoc 或字段注释。
- 对状态流转、事务边界、幂等、锁、缓存、MQ、补偿、兼容性和非显然兜底写“为什么”和“边界”，不翻译代码语法。
- 自解释的简单 getter、setter、私有转发方法和显然字段不写重复名称的废话注释，除非项目工具明确要求。
- 修改行为时同步修改相关注释；移动代码时连同有效注释一起移动。
- 注释语言跟随当前模块稳定约定；JZX 中文模块继续中文，稳定英文模块不强制翻译。
- TODO/FIXME 只有在确实不能本轮收口且项目允许时保留，并写明原因、责任信息或跟踪入口。

### 关键逻辑块注释

方法 Javadoc 只说明公共契约，不能替代方法内部的关键逻辑注释。出现以下任一情况时，在对应代码块前补一条紧邻注释：

- 多步写入或调用顺序影响事务原子性、可见性、重复执行或最终结果。
- 乐观锁、幂等占位、分布式锁、租约、fencing token、条件更新等并发控制。
- MQ / Outbox / Job / 回调的投递顺序、重试、补偿、乱序或至少一次语义。
- 远程调用刻意放在事务外、批量处理用于避免循环 I/O，或存在明确的性能边界。
- 兼容旧数据、权限隔离、安全校验、降级和非显然异常转换。

注释至少说明“为什么这样做”以及“失败或顺序改变会怎样”；不要写“查询数据”“更新状态”这类代码翻译。若本轮 Java 改动没有上述逻辑，在质量记录中明确写“不适用：仅简单 DTO/映射/直线调用”等具体原因。

## 命名、格式和项目工具

- 类、方法、变量和常量使用项目语言约定与准确业务英文，避免孤立的 `data`、`info`、`list`、`map`、`flag`、`type`。
- 方法名表达动作和结果；布尔值表达真实判断语义。
- 不手工发明另一套缩进、换行和 import 顺序。优先执行项目已有 formatter、Spotless、Checkstyle 或 IDE 配置。
- 只格式化本次修改范围；禁止借格式化制造大面积无关 diff。
- 工具通过只能证明其覆盖的机械规则，不能替代接口、数据、事务和业务语义检查。

## 测试与验证

- 从任务完成标准、行为变化和失败影响选择测试层级；优先复用项目已有测试设施和最接近行为边界的测试，不为覆盖率数字制造脆弱用例。
- Bug、核心业务规则、状态迁移、权限、公共接口、SQL/事务和异步副作用变化必须新增或更新自动化测试。Bug 和核心规则在可行时先保存能复现问题的失败结果，再完成 RED → GREEN。
- 纯注释、格式、无行为变化的机械重命名或生成文件可以不新增测试，但仍执行 formatter、编译或静态检查等最小验证。
- 无法合理自动化时允许显式豁免；必须记录不可行原因、残余风险、替代验证、可复跑步骤和后续责任。“用户未要求”或“时间不足”不能单独构成豁免。
- 新增或改变业务规则时覆盖正常、边界、非法状态、重复请求和关键异常分支；测试代码与生产代码遵守同一任务边界，不搭建与风险无关的测试基础设施。
- 修改 Mapper、SQL 或序列化契约时，验证字段映射、空值、枚举、批量参数和兼容性。
- 修改异步、事务或缓存链路时，验证幂等、失败、重试和顺序；无法自动验证的风险明确记录。
- 每个 Txx 先执行与改动最接近的测试，再按影响面扩大到模块或跨仓回归；验证必须基于当前代码新鲜执行。
- 记录命令、工作目录、退出码、输出摘要和对应 Txx；无法执行时说明原因、未覆盖风险和可复跑命令，不把 `git diff --check` 当作编译或测试通过。

## 完成前检查

重新读取最终 Git 差异并逐项确认：

- 改动是否只服务当前任务，是否混入无关重构或格式化。
- 主链路、职责和抽象是否直接清楚，是否存在一次性无效层。
- 接口、DTO、枚举、异常和默认值是否保持契约兼容。
- 参数、身份、权限、敏感信息和数据隔离是否正确。
