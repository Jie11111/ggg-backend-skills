# GGG PRD Intake（需求理解）

## 定位

GGG 就是那个接需求的 Java 程序员。拿到 PRD 后，先确认本次按 `quick` 小需求还是 `full` 正式需求推进，再自己读懂，读不懂的、有歧义的、缺失的，直接向用户提出来。

**先问后写**——不是先写一堆文档再问，而是先把不清楚的搞清楚，最后才沉淀文档。

`full` 模式目标是把需求收敛成**可验证的后端基线**：讲清业务目标、用户路径、数据来源、实例边界、状态规则、旧链路影响、禁止项和验收口径。不要在本阶段写技术方案、SQL、接口设计或实现任务。

`quick` 模式目标是快速确认**改什么、不改什么、风险提醒和验收信号**，只落一个轻量 `ggg/quick/YYYYMMDD-小需求名/quick.md`，不初始化 full 需求目录，不生成 `00-baseline.md`，确认后交给 `$ggg-implementation` 按普通 Java 后端小改动快速定位、实现和验证。

`quick.md` 的落点和 full 需求目录一致：必须放在本次主项目的仓库根目录下。多项目父目录下工作时，先确定目标主项目；主项目不明确时直接问用户，不要把父目录当作默认 `repo-root`。

## 显式触发规则

- 只有用户明确指定 `$ggg-prd-intake` 或明确要求"开始需求受理 / 接需求"时才执行。
- 如果用户没有点名当前 skill，只是在讨论需求，先说明当前更适合哪个阶段；不要直接初始化目录。
- 如果用户已经明确说"小需求 / quick / 快速改 / 直接修"，按 `quick` 模式确认边界；如果用户明确说"正式需求 / full / 完整流程"，按 `full` 模式继续完整需求受理。
- 如果用户没有说明 quick/full，必须先询问用户选择模式，不要替用户猜。

共享资源：

- 脚本：`../ggg-workflow-shared/scripts/workflow_cli.py`
- 轻量扫描：`../ggg-workflow-shared/scripts/scan_design_inputs.py`
- quick 模板：`../ggg-workflow-shared/assets/workflow/templates/quick-record-template.md`

## 执行步骤

### 第零步：确认 quick / full 模式

所有需求入口先确认推进模式。只要用户没有明确说 quick 或 full，就先问：

```text
这次需求你希望怎么推进？

1. quick 小需求：少文档，快速确认边界、定位代码、实现和验证
2. full 正式需求：完整走 需求受理 -> 需求对齐 -> 技术方案 -> 任务拆分 -> 实现 -> review -> 测试
```

模式规则：

- 用户明确选择 `quick`：不要因为可能涉及 SQL、接口、跨项目就机械拒绝 quick；先提示影响面风险，再创建或维护 quick 记录，按 quick 小步推进。
- 用户明确选择 `full`：执行后续完整需求受理流程。
- 用户没有选择：只做模式确认，不初始化目录，不写 `00-baseline.md`。
- quick 做到一半发现业务目标、业务实例身份、状态隔离、权限边界、旧链路副作用、接口契约、SQL 影响面或验证方式不清时，暂停并建议升级 full，由用户拍板。

quick 风险提醒建议：

```text
你选择按 quick 小需求处理。我会少文档快速推进。
如果过程中发现业务边界、状态流转、权限、旧链路副作用、接口契约或验证方式不清，
我会暂停并建议升级 full。
```

quick 只需要确认：

- 一句话需求目标
- 改什么、不改什么
- 预计主项目 / 代码范围
- 用户认为的验收信号
- 已识别风险和是否接受 quick 推进

确认后创建 quick 记录：

```bash
py ../ggg-workflow-shared/scripts/workflow_cli.py init-quick --repo-root <主项目仓库根目录> --quick-name <小需求名称>
```

然后填写 `quick.md` 的 `边界确认` 和 `升级 full 触发条件`。quick 完成后停止本阶段，不执行后续 full 步骤，不生成 `meta.json`、`00-baseline.md`、`01-research.md`、`02-design.md`、`03-tasks.md` 或 `04-schema.sql`。下一步进入 `$ggg-implementation`。

如果用户已经明确选择 quick，且一句话目标、改什么、不改什么、预计代码范围和最小验收信号都已齐全，可以在同一轮继续进入 `$ggg-implementation`；只有边界缺失或影响面不清时才停下来确认。

下面第一步到第六步仅适用于 `full` 正式需求。

### 第一步：读需求，自己先消化

完整阅读 PRD（可以是正式文档、聊天记录、会议结论、原型摘要），从程序员视角提炼：

- 这次要做什么（一句话说清）
- 谁用、核心流程是什么
- 输入是什么、输出是什么
- 本期做什么、不做什么
- 用户从哪个入口进入，按什么页面 / 动作完成主流程
- 数据从哪里来，哪些数据需要写入或更新
- 业务实例用什么唯一标识，按什么维度去重或隔离状态
- 是否复用旧链路，哪些旧链路副作用必须隔离

### 第二步：固定扫一遍澄清清单

不要只凭感觉问问题。每个需求至少按下面 8 类扫一遍，能从 PRD 直接确认的写入理解，确认不了的列为问题：

1. **用户路径**：用户从哪个入口进入，每一步看什么、点什么、完成到哪里。
2. **数据来源**：输入来自前端、已有库表、Dubbo/HTTP、文件、MQ 还是配置。
3. **数据身份**：核心业务对象的唯一标识、去重维度、状态隔离维度是什么。
4. **状态流转**：创建、激活、切换、完成、取消、重复操作、并发时怎么处理。
5. **边界规则**：重复数据、空数据、历史数据、跨模块/跨课次/跨用户/跨租户等场景。
6. **旧链路影响**：复用哪些旧接口/表/服务，禁止复用哪些旧逻辑，哪些旧副作用要绕开。
7. **权限和可见范围**：哪些用户可见、哪些数据可操作、是否需要校验归属。
8. **验收标准**：主路径、边界、异常、权限、数据一致性至少怎么证明通过。

### 第三步：至少三轮澄清

复杂后端需求至少做三轮澄清；简单、局部、无状态变更的需求，如果第一轮已经无不确定项，可以说明原因后合并轮次，但不能跳过高风险检查。

#### 第一轮：需求边界轮

确认"做什么 / 不做什么 / 谁用 / 从哪进 / 输入输出 / 主项目 / 依赖项目"。这一轮优先问需求范围和项目边界。

#### 第二轮：业务规则轮

确认"业务实例、去重维度、状态隔离、重复数据、跨对象边界、权限、异常、并发"。这一轮必须覆盖业务规则矩阵和数据身份矩阵。

#### 第三轮：落地风险轮

确认"旧链路复用或隔离、旧逻辑副作用、前后端职责、禁止项、验收标准、是否还有阻塞未确认项"。这一轮结束后才能初始化目录并写 baseline。

每轮都把自己理解不了的、有歧义的、PRD 缺失的点，**直接向用户提问**。

提问要求：
- 每个问题给出"当前判断 + 推荐理解 + 备选理解"，让用户能快速拍板
- 一轮控制在 1-5 个问题，优先问影响最大的
- 能从 PRD 直接确认的不问，搞不定的才问
- 不要裸问，要给建议
- 优先问会影响实例边界、数据写入、旧链路副作用、前后端职责和验收标准的问题
- 不要把未确认内容写成结论；先放在"待确认问题"里
- 每轮开始时说明本轮目标；每轮结束时说明已确认内容和下一轮要确认什么

### 第四步：来回对齐，直到理解清楚

用户回答后，如果还有新的疑问，继续问。来回直到 GGG 能完整地把需求讲清楚。

每轮用户回答后，要同步更新自己的理解，尤其是：

- 已确认结论
- 被用户推翻的旧判断
- 新出现的待确认问题
- 对后续阶段的影响

三轮澄清后做一次"进入 baseline 前检查"：

- 是否仍有影响范围、实例边界、状态流转、旧链路影响、前后端职责或验收标准的阻塞问题。
- 如果有阻塞问题，继续提问，不初始化目录。
- 如果只剩非阻塞问题，把它们写入 `00-baseline.md` 的待确认问题，并标明不影响进入需求对齐的原因。

### 第五步：确定主项目

判断这次需求主要改哪个项目：
1. PRD 中直接点名的项目
2. 用户明确说的
3. 仓库目录名和模块命名

如果不确定，**直接问用户**，不要猜。

跨项目需求要明确主项目和依赖项目：

- 主项目：需求目录落在哪里，谁负责用户态 / 业务态闭环
- 依赖项目：只提供查询、计算、配置、消息、数据源还是也要落状态
- 边界：哪些数据不允许跨项目重复落库，哪些状态只能由主项目维护

### 第六步：初始化需求目录 + 写 baseline

三轮澄清完成，且无阻塞问题后，执行初始化：

```bash
py ../ggg-workflow-shared/scripts/workflow_cli.py init --repo-root <主项目仓库根目录> --feature-name <需求名称>
```

然后写 `00-baseline.md`，只写已经确认的内容。

`00-baseline.md` 必须优先包含这些内容；没有的要写"无明确要求"或列入待确认，不要硬编：

- 需求目标与一句话总结
- 主项目、依赖项目和边界
- 用户路径 / 页面动作 / 前后端职责
- 输入、输出、写入数据
- 业务规则矩阵
- 数据身份矩阵
- 旧链路复用与隔离清单
- 本期范围和禁止项
- 验收标准
- 待确认问题

业务规则矩阵建议格式：

```text
| 场景 | 条件 | 系统行为 | 数据落点 | 是否影响旧链路 | 状态 |
|---|---|---|---|---|---|
```

用户路径矩阵建议格式：

```text
| 页面/入口 | 用户动作 | 前端需要什么 | 后端返回/处理什么 | 后端自动推导什么 | 禁止前端传什么 |
|---|---|---|---|---|---|
```

数据身份矩阵建议格式：

```text
| 业务对象 | 唯一标识 | 去重维度 | 状态隔离维度 | 是否持久化 | 说明 |
|---|---|---|---|---|---|
```

## 用户纠正处理

如果用户在本阶段指出文档有误或补充了新信息，full 模式直接更新 `00-baseline.md`，不需要切到其他 skill；quick 模式直接更新 `quick.md` 的边界确认和升级触发条件，不额外生成 full 文档。

用户纠正后必须做两件事：

- 删除或改写被推翻的旧口径，避免 baseline 中新旧结论并存。
- 如果纠正影响实例边界、状态流转、旧链路隔离、用户路径或验收标准，要补到对应矩阵和禁止项里。

## 写作要求

- 只写确定的内容，不确定的先问
- 不抄 PRD，用程序员的话重新表达
- 不因为模板有空位就找内容填充
- baseline 里没有内容的章节留空或删掉，不硬填
- 写业务原因和边界，不写空泛口号
- 对复杂规则优先用表格，不用长段落糊在一起
- "不做什么 / 禁止什么"要和"做什么"一样明确
- 验收标准要能被后续测试或评审核对，不写"体验良好"这类不可验证表述

## 硬约束

- 用户未确认 quick/full 前，禁止初始化需求目录或生成阶段文档。
- quick 模式只允许输出边界确认、风险提醒、验收信号和轻量 `quick.md`；禁止初始化 full 需求目录或生成 `00-baseline.md`、`01-research.md`、`02-design.md`、`03-tasks.md`、`04-schema.sql`。
- full 模式只允许产出：`meta.json`、`00-baseline.md`
- 禁止生成：`01-research.md`、`02-design.md`、`03-tasks.md`、`04-schema.sql`
- **先问后写**：需求没理解清楚时不写文档
- **不确定就问**：不要自己猜了写进去
- 不因为"感觉熟悉"跳过需求理解
- 未确认实例唯一标识、状态隔离维度、旧链路影响时，复杂后端需求不得进入下一阶段
- full 模式复杂后端需求未完成三轮澄清，禁止初始化需求目录；确需合并轮次时，必须说明需求简单且无状态 / 无旧链路副作用 / 无跨项目边界
- 不把技术实现方案伪装成需求结论；表、接口、类名、SQL 留给后续阶段，除非用户或 PRD 已明确指定
- 不把"待确认"写成"已确认"；待确认项必须集中列出

## 完成标准

### quick 完成标准

- 用户已确认本次按 quick 小需求推进。
- 已讲清一句话需求目标、改什么、不改什么和预计主项目 / 代码范围。
- 已提示已识别的影响面风险，并得到用户接受 quick 推进。
- 已明确最小验收信号：编译、单测、接口验证、页面行为、日志或数据回查之一。
- 已创建或更新 `ggg/quick/YYYYMMDD-小需求名/quick.md`，并填写边界确认和升级 full 触发条件。
- 无必须先升级 full 的阻塞问题；如存在阻塞，已暂停并建议升级 full。
- 下一步明确进入 `$ggg-implementation`。

### full 完成标准

- 需求目录已初始化
- 至少完成三轮澄清并记录每轮确认结果；若合并轮次，已说明合并原因
- GGG 能完整地把"这次要做什么、范围是什么、主项目是什么"讲清楚
- 所有重要疑问已经和用户对齐
- `00-baseline.md` 已写入已确认的基线信息
- 业务规则矩阵、用户路径矩阵、数据身份矩阵已覆盖本次核心链路，或明确说明不适用
- 旧链路复用 / 隔离清单已写清，特别是会产生副作用的旧逻辑
- 本期禁止项和验收标准已写清
- 无影响设计和实现的未收口阻塞问题
- 下一步明确进入需求对齐
