# GGG Requirement Alignment（需求对齐）

## 定位

GGG 就是那个程序员，需求理解完了，开始看代码确认。看看主入口在哪、现有逻辑能不能复用、有没有和需求冲突的地方。

**只确认和需求直接相关的代码**——不做全仓扫描，不找"类似逻辑"。

本阶段不是凭感觉扫代码，而是拿 `00-baseline.md` 逐条验证代码事实：用户路径能不能走通，业务规则有没有现有承载，数据身份和状态维度是否匹配，旧链路有没有副作用，哪些能力能复用、哪些必须新增。

代码对齐不能假装全知。所有结论都必须说明证据来源、覆盖范围和置信度；**未找到不等于没有**，只能写"在当前证据范围内未发现"。

## 显式触发规则

- 只有用户明确指定 `$ggg-requirement-alignment` 或明确要求"开始需求对齐 / 看代码确认"时才执行。
- 缺少 `meta.json` 或 `00-baseline.md` 时，先引导用户使用 `$ggg-prd-intake`。

共享资源：

- 脚本：`../ggg-workflow-shared/scripts/workflow_cli.py`
- 轻量扫描：`../ggg-workflow-shared/scripts/scan_design_inputs.py`

## 前置条件

- 已存在 `meta.json` 和 `00-baseline.md`
- 需求已经理解清楚（prd-intake 阶段完成）
- `00-baseline.md` 中关键用户路径、业务规则、数据身份、旧链路影响、禁止项和验收标准已经收口；如果这些内容缺失，先回到 `$ggg-prd-intake`

## 执行步骤

### 第一步：推进阶段

```bash
py ../ggg-workflow-shared/scripts/workflow_cli.py to-alignment --feature-dir <需求目录>
```

### 第二步：从 baseline 生成代码验证清单

先读取 `00-baseline.md`，提取本阶段必须验证的内容：

- 用户路径 / 页面动作 / 前后端职责
- 业务规则矩阵
- 数据身份矩阵
- 旧链路复用与隔离清单
- 本期禁止项
- 验收标准
- 主项目和依赖项目边界

形成验证清单，不要直接开始搜代码。建议格式：

```text
| baseline 条目 | 需要验证的代码位置 | 验证状态 | 代码事实 | 结论 | 风险 |
|---|---|---|---|---|---|
```

验证状态只使用：

- 已验证：入口、核心调用、数据结构或依赖能力已经有代码证据闭合。
- 部分验证：主链路有证据，但配置、运行时行为、旁路任务或依赖项目仍有缺口。
- 未覆盖：当前证据没有覆盖到，不能写成确定结论。
- 阻塞：缺口会影响技术方案判断，必须先向用户确认或继续查证。

验证清单要覆盖：

- 每条关键业务规则
- 每个核心业务对象的唯一标识、去重维度、状态隔离维度
- 每个禁止复用的旧链路
- 每个跨项目依赖
- 每个验收标准对应的代码能力

### 第三步：自动确认 CodeGraph 目标项目和索引新鲜度

需要用 CodeGraph 查代码时，先自动识别本次涉及的子项目，再检查索引状态；只有自动判断失败、索引不可用、需要初始化/更新但没有明确授权时，才向用户确认。不要一开始就问用户“有没有初始化”。

自动识别顺序：

1. 优先读取 `00-baseline.md` 的主项目、依赖项目、主项目判断依据。
2. 结合 AGENTS.md 或工作区约定的子项目路径，例如 `poseidon`、`reasearch-hub`、`jzx-server`、`2b`。
3. 检查子项目目录是否存在，不能把多项目父目录当默认 CodeGraph 项目。
4. 对每个候选子项目显式传入 `projectPath` 执行 `codegraph_status`。
5. 如果 CodeGraph 工具不可用，再检查目标子项目下是否存在 `.codegraph/`，并降级说明证据风险。

自动探测输出建议：

```text
本次预计使用 CodeGraph 查询这些子项目：
- reasearch-hub: d:\towerProject\jzx-tob\reasearch-hub，status 已检查，可用
- poseidon: d:\towerProject\jzx-tob\poseidon，status 显示 pending sync，pending 文件将直接读取最新内容
```

如果目标子项目未初始化：

- 不要在父目录初始化。
- 默认提示用户对目标子项目执行初始化，例如 `codegraph init <子项目路径>` 或当前 CodeGraph 环境推荐的初始化命令。
- 只有用户已经明确授权“可以自动初始化/更新 CodeGraph”，或本次请求本身就是初始化/更新索引，才可以对确定的子项目执行初始化。
- 自动初始化前必须再次输出目标绝对路径，确认不是工作区父目录。
- 如用户要求先继续，只能用 `rg` / 文件读取做降级分析，并在 `01-research.md` 明确标注证据置信度下降。

如果 CodeGraph 已初始化但 status 显示 pending sync、索引滞后，或工具响应提示"文件已编辑但索引未同步"：

- 不要基于旧索引对 pending 文件下确定结论。
- 对 pending 文件直接用文件读取获取最新内容。
- 对未 pending 的文件仍可继续信任 CodeGraph。
- 在 `01-research.md` 的覆盖度里标注哪些结论来自 CodeGraph，哪些来自直接读取兜底。
- 如果 pending 范围影响主链路或复用边界，优先等待索引同步后重新跑 `codegraph_status`；如环境支持显式刷新/更新，只有在目标子项目明确、不会落到父目录、且用户已授权自动更新时才执行。
- 如果无法等待或更新，只能直接读取 pending 文件并在覆盖度里标注“CodeGraph pending 文件已用直接读取兜底”。

### 第四步：优先使用 CodeGraph 做结构化代码验证

优先使用 CodeGraph 查询代码事实。使用 CodeGraph 前必须先判断目标代码属于哪个子项目，并显式传入对应 `projectPath`，不要在多项目父目录查默认索引。

常用子项目：

- `poseidon`：`d:\towerProject\jzx-tob\poseidon`
- `reasearch-hub`：`d:\towerProject\jzx-tob\reasearch-hub`
- `jzx-server`：`d:\towerProject\jzx-tob\jzx-server`
- `2b`：`d:\towerProject\jzx-tob\2b`

CodeGraph 使用顺序：

0. `codegraph_status`：确认目标子项目索引可用且无影响本次范围的 pending sync。
1. `codegraph_search`：定位明确类、方法、DTO、枚举、Mapper、Facade。
2. `codegraph_explore`：理解入口到核心逻辑的主链路，优先一次提问覆盖相关符号。
3. `codegraph_callers` / `codegraph_callees`：确认调用方、被调方、旁路副作用。
4. `codegraph_impact`：判断复用或修改影响面。
5. 仅当 CodeGraph 覆盖不到 SQL XML、配置、字符串常量、脚本或未索引文件时，再用 `rg` / 文件读取补充。

如果目标子项目尚未初始化 CodeGraph，只提示或在授权后对该子项目执行初始化，不要在父目录初始化。

### 第五步：从主项目入口开始看代码

按固定顺序查代码，只看和需求直接相关的：

1. Controller / Job / MQ / 定时任务入口
2. Service / Manager
3. Dubbo Consumer / HTTP Client
4. Facade / Provider
5. Mapper / SQL / 表结构

**相关性判定**：每查到一段代码，先判断：
- 是不是本次需求主链路上一定会经过的？→ 记录
- 是不是 PRD 或用户明确提及的？→ 记录
- 以上都不是，只是"看起来类似"？→ **问用户确认后再决定是否记录**

### 第六步：强制检查旧链路副作用

每个复杂后端需求都要输出旧链路副作用清单。不要只看新入口是否能写出来，还要看旧逻辑是否会误伤新需求。

重点检查：

- 全局状态维度是否和新需求冲突，例如 `userId + ptId`、`userId + date`。
- 旧接口是否有自动完成、自动重排、自动推荐、自动同步、自动补偿逻辑。
- 旧推荐池、旧排序、旧候选来源是否会混入新链路。
- Redis / MQ / ES / 定时任务 / 事务后通知 / 分布式锁是否会产生旁路影响。
- 老接口默认行为是否会在新链路中被误调用。
- 缓存、配置、开关、灰度逻辑是否会改变可见范围或数据来源。

旧能力必须分级，不允许只写"复用"：

```text
| 旧能力 | 代码位置 | 现有语义 | 反向影响范围 | 与本需求关系 | 结论 | 说明 |
|---|---|---|---|---|---|---|
```

结论只使用：

- 可直接复用：语义、维度、副作用完全匹配。
- 可扩展复用：入口或结构可用，但需补字段、分支或校验。
- 只可参考：实现思路可参考，但不能接入旧逻辑。
- 禁止复用：语义冲突或副作用会破坏需求。
- 必须新增：旧链路无法承载本需求。

凡是准备复用或改造旧代码，必须做一次反向影响检查：

- 用 `codegraph_callers` 确认旧方法还有哪些接口、任务、MQ、Facade 调用。
- 用 `codegraph_impact` 判断改动会影响哪些旧场景、旧数据、旧缓存或旧推荐链路。
- 如果反向影响没有闭合，复用结论最高只能写"可扩展复用（待确认）"或"只可参考"。
- 如果发现旧场景会被新语义污染，直接降级为"禁止复用"或"必须新增"。

### 第七步：对齐跨项目依赖

如果 baseline 涉及多个项目，必须单独验证：

- 主项目负责哪些用户态 / 业务态闭环。
- 依赖项目已有 Facade / HTTP / MQ 能力是否够用。
- 依赖 DTO 字段是否支持本需求的用户路径、业务规则和验收标准。
- 依赖项目是否允许落状态；如果 baseline 明确不落状态，不能在对齐结论里建议落状态。
- 是否需要新增 Dubbo / HTTP 方法，或只需复用现有查询能力。

### 第八步：标注证据覆盖度和置信度

每个关键结论都要标注证据覆盖度、证据等级和置信度，不允许把推断写成事实。

置信度分级：

- 高：入口、调用链、核心 Service、Mapper / 表结构或依赖 Facade 均已闭合，且无明显动态配置缺口。
- 中：源码主链路已验证，但配置、运行时开关、MQ / 定时任务 / 缓存 / 反射 / 前端调用仍有未覆盖项。
- 低：只有关键词命中、局部代码片段或推断，必须继续确认。

覆盖度表建议：

```text
| 结论ID | 证据来源 | 证据等级 | 已覆盖代码 | 未覆盖范围 | 运行时证据缺口 | 置信度 | 后续确认方式 |
|---|---|---|---|---|---|---|---|
```

证据等级只使用：

- 代码已证实：有具体文件 / 方法 / 调用链 / 表字段证据，且证据 ID 可追溯。
- 编译已证实：有编译命令、退出码或关键输出证据。
- 接口已证实：有请求、响应、业务成功标识或错误证据。
- 数据已证实：有 MySQL / ES / Redis / 日志回查证据。
- 推断：有依据但未闭合，只能进入风险或待确认，不能作为高置信方案依据。
- 未覆盖：当前证据没有覆盖到，不能写成确定结论。
- 阻塞：缺口影响技术方案判断，必须先确认。

关键结论必须分配 `C1/C2/...` 结论 ID，并引用 `E1/E2/...` 证据 ID；没有证据 ID 的内容只能写入“推断 / 未覆盖 / 阻塞”，不能流入技术方案。

必须明确写出低覆盖区域，例如：

- CodeGraph 未初始化或索引滞后。
- CodeGraph 已初始化但存在 pending sync，且 pending 文件影响本次范围。
- SQL XML、Nacos 配置、脚本、枚举字符串、前端调用未覆盖。
- Dubbo 动态路由、反射、SPI、MQ 消费、定时任务、缓存刷新无法静态确认。
- 依赖项目没有查全。
- 需要接口验证、日志、运行时配置或测试数据才能确认。

运行时证据缺口只用轻量分类，不展开成长清单：

- 配置确认：Nacos、开关、灰度、路由、环境差异。
- 接口验证：需要真实 token、请求参数、前端入口或联调环境确认。
- 日志/运行时确认：需要看日志、链路追踪、MQ 消费、定时任务、缓存刷新。
- 测试数据确认：需要构造用户、班级、课程、学习包、历史状态等数据。
- 用户口径确认：代码事实和 baseline 不一致，或旧规则是否沿用需要业务确认。

### 第九步：建立结论账本（Claim Ledger）

在 `01-research.md` 中固定维护结论账本。凡是会影响技术方案、SQL、接口契约、任务拆分或旧链路复用的判断，都必须进入结论账本。

建议格式：

```text
| 结论ID | 关键结论 | 结论类型 | 证据ID | 证据等级 | 置信度 | 未覆盖范围 | 若结论错误的影响 | 后续确认方式 |
|---|---|---|---|---|---|---|---|---|
```

结论账本规则：

- `Cxx` 必须引用至少一个 `Exx` 证据 ID；证据 ID 必须能在“代码证据索引”中找到。
- 文件类 `Exx` 证据的“位置”必须写仓库根相对路径或绝对路径加行号，例如 `src/main/java/com/demo/ReportController.java:42`；不能只写类名、方法名或“已确认”。
- 文件类 `Exx` 证据必须能通过 validate 检查文件存在和行号范围；无法落到文件行号的运行时、接口、日志、DB/ES/Redis 证据，类型必须明确写为运行时 / 接口 / 日志 / 数据证据，并说明证据来源。
- 高置信结论必须有具体代码证据或运行证据，不能只来自关键词命中、经验判断或“看起来类似”。
- 低置信或推断类结论必须写清未覆盖范围和后续确认方式，不能被下游方案当作确定事实。
- 旧能力“可直接复用 / 可扩展复用 / 禁止复用 / 必须新增”都必须有 `Cxx`，并绑定反向影响检查证据。
- 如果结论错误会影响 SQL、接口契约、权限、状态隔离、数据一致性或旧链路副作用，且证据不闭合，必须标为阻塞。

### 第十步：有问题就提出来

代码和需求有冲突的、不确定的，直接向用户提问。每个问题带上：
- 当前判断
- 推荐方案
- 备选方案

必须提问的阻塞类型：

- 主链路入口不清。
- 实例唯一标识不清。
- 状态隔离维度不清。
- 旧链路副作用未识别或无法规避。
- 可复用 / 禁止复用边界不清。
- 跨项目依赖能力不清。
- 关键表、字段、DTO、枚举是否支持不清。
- 代码证据覆盖度不足，且会影响技术方案判断。
- 静态代码结论和 baseline 业务规则冲突。

### 第十一步：发现新问题要回写上游文档

对齐阶段发现的新问题不能只写在 `01-research.md`。如果它改变了需求理解或上游基线，必须回写：

- 影响需求范围、用户路径、业务规则、数据身份、旧链路隔离、禁止项、验收标准 → 更新 `00-baseline.md`。
- 只影响代码事实、复用边界、实现落点、依赖能力 → 更新 `01-research.md`。

回写时要删除或改写旧口径，避免 `00-baseline.md` 和 `01-research.md` 新旧结论并存。

### 第十二步：写结论

把确认的结论写入 `01-research.md`，更新 `00-baseline.md` 的代码现状和差异章节。

结论要写清：
- baseline 验证清单和验证结果
- 每个验证点的状态：已验证 / 部分验证 / 未覆盖 / 阻塞
- 主链路怎么走
- 哪些现有代码可以复用
- 哪些地方需要新增或改造
- 旧链路副作用清单
- 旧能力复用或改造的反向影响检查结果
- 数据身份和状态维度对照
- 复用性分级
- 跨项目依赖能力
- 代码证据覆盖度、运行时证据缺口和置信度
- 结论账本：每个关键结论的 `Cxx`、证据 ID、证据等级、置信度、未覆盖范围和错误影响
- 进入技术方案前的阻塞问题和非阻塞待确认项
- 残余风险和后续确认方式
- 关键代码证据

推荐 `01-research.md` 结构：

```text
1. Baseline 验证清单
2. 主链路代码事实
3. 旧链路副作用清单
4. 数据身份和状态维度对照
5. 可复用 / 可扩展 / 只参考 / 禁止复用 / 必须新增
6. 旧能力反向影响检查
7. 跨项目依赖能力
8. 代码证据覆盖度、运行时证据缺口和置信度
9. 结论账本（Claim Ledger）
10. 进入技术方案前阻塞问题
11. 残余风险和后续确认方式
12. 代码证据索引
```

### 第十三步：同步、校验和残余风险确认

```bash
py ../ggg-workflow-shared/scripts/workflow_cli.py sync-meta --feature-dir <需求目录>
py ../ggg-workflow-shared/scripts/workflow_cli.py validate --feature-dir <需求目录>
```

validate 通过后才能进入技术方案。

validate 通过不等于可以进入技术方案；还必须确认 `01-research.md` 没有影响方案的阻塞问题。

进入下一阶段前必须列出残余风险：

- 已确认：哪些链路、表、接口、依赖已经闭合。
- 未覆盖：哪些代码、配置、运行时行为没有覆盖。
- 非阻塞风险：为什么不影响进入技术方案，并标注属于配置确认、接口验证、日志/运行时确认、测试数据确认还是用户口径确认。
- 阻塞风险：必须继续向用户确认，不能进入技术方案。
