# GGG Requirement Alignment（需求对齐）

## 目标和边界

本阶段拿已确认的 `00-baseline.md` 验证真实代码，不重新发明需求，也不直接设计方案。最终回答五个问题：

1. 用户闭环在现有系统中怎样运行。
2. 业务事实、关联产物和状态由哪里承载。
3. 失败重试、再次操作和异步旁路是否改变语义。
4. 哪些能力可复用、需改造、只能参考或禁止复用。
5. 新增或变化的状态、枚举、类型码和字段语义被哪些场景生产、读取、过滤、统计或传播。
6. 哪些结论已闭合，哪些非阻塞缺口留待后续验证。

只看与需求直接相关的代码。可以按需读代码、SQL、配置、日志或接口结果，但禁止无边界全仓扫描，也不能把“看起来类似”写成正式结论。

技术事实由 AI 用代码和运行证据查证，不让用户判断代码相关性。只有业务意图无法从需求材料确定时才问用户；实现方式未定但业务口径清楚时转交技术方案。

## 执行流程

### 1. 从 baseline 建立验证清单

先读取 `00-baseline.md`，逐条提取：用户闭环、关联产物不变量、业务规则、数据身份、旧链路复用与隔离、禁止项、验收标准、主项目和依赖项目。

Schema v4 baseline 中每条需要代码验证的业务行都有唯一 `Bxx`。必须把全部 `Bxx` 逐项写入 `01-research.md` 的 Baseline 验证清单，不得合并、省略或增加不存在的 Bxx。验证状态只允许：

- `已验证`：入口、核心调用和数据承载有证据闭合。
- `部分验证`：主链路已确认，但存在不影响方案判断的配置或运行时缺口。
- `未覆盖`：当前证据范围未覆盖，不能写成确定结论。
- `阻塞`：缺口会影响方案判断；必须引用唯一疑问账本的 `Qxx`，且不能通过本阶段门禁。

适用时必须验证关联产物、数据承载和失败重试/再次操作；主链路每个固定项都要填写真实结论。第 3～7 节至少有一条真实业务行，明确不适用时统一写“`不涉及：具体原因`”，不保留空标签或空模板行。

### 2. 确认真实代码范围

1. 从 baseline 的主项目和依赖项目定位目录。
2. 读取适用的 `AGENTS.md` 和工作区约定。
3. 多项目工作区只选择已确认的项目或模块，不把共同父目录当默认范围。
4. 开始 CodeGraph 调研前必须提醒用户初始化或刷新；每个目标项目分别给出命令，不能对多项目共同父目录执行。
5. 用户完成后由 AI 执行 `codegraph status "<目标项目目录>"` 验证状态，再开始 CodeGraph 查询。

提醒格式固定为：

```text
CodeGraph 调研前提醒
- 当前状态：<项目> 未初始化 / 建议刷新后再调研
- 请执行：codegraph init "<目标项目目录>"
  或：codegraph sync "<目标项目目录>"
- 完成后请回复“已完成”；暂不刷新可回复“跳过刷新”
```

- 目录不存在 `.codegraph/` 时只提供 `codegraph init "<目标项目目录>"`。
- 目录已经初始化时提供 `codegraph sync "<目标项目目录>"`，不重复初始化。
- 这只是工具状态提醒，不进入业务 Qxx 疑问账本。
- 用户选择跳过、命令失败或 CodeGraph 不可用时，不使用旧索引下结论；改用直接读取和 `rg` 查证，并降低对应结论置信度。

正式证据优先写仓库根相对路径加行号。跨仓证据无法稳定表达时可写绝对路径加行号，但文档必须同时注明项目；Skill、示例和模板中不得固化个人机器路径。

### 3. 优先做结构化代码验证

存在 `.codegraph/` 时，优先使用当前环境可用的 `codegraph_explore`、`codegraph explore` 或等价能力定位符号、调用路径和影响面；需要时使用环境提供的 callers、callees、impact、search、status 等等价能力，不假设固定子命令名。

CodeGraph 不覆盖 SQL XML、配置、脚本、字符串或未索引文件时，再用 `rg` 和直接读取补证。

只有 CodeGraph 无法定位候选入口，且范围已经缩小到明确项目或模块时，才执行有界扫描：

```bash
python3 "${CODEX_HOME:-$HOME/.codex}/skills/ggg-workflow-shared/scripts/workflow_cli.py" \
  scan-design-inputs --scope-root <目标项目或模块目录> \
  --max-files 500 --max-depth 8 --limit 20
```

扫描器规则：

- `max-files` 和 `max-depth` 限制实际纳入文件，`limit` 只限制每类展示行数。
- 文件和目录符号链接不跟随，扫描内容不得逃逸 `scope-root`。
- 禁止扫描包含 `ggg/features` 的仓库根目录，无绕过参数。
- 配置扫描只能输出配置类别和文件位置，禁止输出、保存配置值。
- 扫描被截断时缩小范围后重扫；候选必须继续核对调用链，不能直接写成 `Exx` 或 `Cxx`。

### 4. 沿用户闭环查证

从实际入口出发，按调用链查 Controller / Job / MQ、Service / Manager、Consumer / Client、Facade / Provider、Mapper / SQL / 表结构。不要只按分层逐文件浏览。

每条主链至少回答：

- 一次触发经过哪些同步和异步节点。
- 产生几个业务实例及关联产物。
- 关键业务事实在哪里首次产生、保存和读取。
- 失败重试或再次操作是否重新计算或污染旧状态。
- Redis、MQ、ES、定时任务、事务后通知、锁、缓存、配置和灰度是否形成旁路影响。

对相关链路额外检查静态调用图容易遗漏的框架隐式行为：

- AOP / Aspect：切点、通知顺序、鉴权、日志、幂等、锁和事务增强。
- Filter / Interceptor / Guard / Middleware：请求改写、上下文注入、权限、短路和异常转换。
- 动态 Bean：`@Conditional`、`@Profile`、`@Primary`、`@Qualifier`、工厂、SPI 和运行时实现选择。
- 事务边界：`@Transactional`、传播级别、回滚规则、事务后事件和提交后通知。
- 异常处理：全局异常处理、重试、熔断、降级、补偿以及异常被吞掉后的实际结果。
- 生成代码：注解处理器、生成客户端/DTO、代理类和构建期生成物对真实调用关系的影响。

只检查与本需求入口或数据流有证据关系的隐式机制；不适用时写明原因。静态源码无法证明运行时选择时，记录到对应 `Cxx` 的运行时证据缺口；影响方案判断时转成阻塞 Qxx。

相关性必须由需求材料、调用关系或数据流证明。最终没有证据证明相关的代码留在内部候选，必要时只记录“当前证据范围内未发现”。

### 4.1 反向验证共享状态、枚举和类型语义影响面

新增或修改状态、枚举值、类型码、业务阶段、场景标识，或者改变既有字段语义时，不能只从需求入口沿调用链向下查。必须以共享语义及其数据载体为中心，补做“生产方 → 持久化/传播 → 全部消费场景”的双向验证。

先确认权威载体和真实值：Java 枚举/常量、数据库字段及存储值、DTO/Facade/HTTP 字段、MQ 事件、Redis/ES/缓存字段。搜索不能只用枚举类名，还要覆盖字段名、序列化值、数字码、历史魔法值和 SQL/XML。

再检查与该语义有证据关系的读取和传播位置：

- `switch`、`if/else`、三元表达式、EnumSet、Map、白名单和黑名单。
- SQL/XML 中的等值、不等值、`IN/NOT IN`、`CASE WHEN`、聚合、分组、排序和过滤。
- 学习报告、统计、异常检测、错题、日清等衍生业务场景。
- Job、补偿、MQ Consumer、监听器、异步任务和事务后事件。
- RPC/HTTP/DTO/JSON 序列化，以及 Redis、ES、缓存和前端状态映射。
- 历史数据、旧调用方、默认分支和未知枚举值的兼容行为。

在 `01-research.md` 的“共享状态、枚举和类型语义影响矩阵”中记录实际检索范围，并按消费场景逐行写清当前规则、新语义预期、代码改动结论、Exx/Cxx 和验证缺口。代码改动结论只允许 `需改 / 无需改 / 仅运行验证 / 阻塞`；这是取证后的场景结论，不使用 `Must change / May change / No code change` 仓库分类。

每个真实场景必须引用一个类型为“共享语义影响”的 Cxx。业务预期不明确时进入 Qxx 并阻塞，不得根据当前 `default` 分支或旧 SQL 反推需求；静态代码无法证明运行时选择时写“仅运行验证”，并在 Cxx 中保留运行时证据缺口。只有记录了实际检索边界并闭合所有已发现的消费场景，才能写“当前证据范围内影响面已闭合”，不得声称穷尽了未纳入范围的仓库或运行时消费者。

本需求不涉及上述共享语义变化时，影响面判断写“`不涉及：具体原因`”，不机械填充矩阵。

### 5. 验证数据身份和旧链路副作用

对核心业务对象核对唯一标识、去重维度、状态隔离维度和现有字段。对准备复用或改造的旧能力，核对所有调用方和影响面，确认旧接口默认行为、推荐池、排序、缓存、配置或补偿逻辑不会污染新语义。

复用等级只允许：

- `可直接复用`：语义、维度和副作用完全匹配。
- `可扩展复用`：结构可用，但需补字段、分支或校验。
- `只可参考`：思路可参考，不能接入旧逻辑。
- `禁止复用`：语义冲突或副作用破坏需求。
- `必须新增`：旧链路无法承载。

每个复用判断都进入唯一 Claim Ledger，并引用反向影响证据。影响面未闭合时不能写“可直接复用”。

### 6. 对齐跨项目依赖

涉及多个项目时分别确认：主项目负责的闭环、依赖项目已有 Facade / HTTP / MQ 能力、DTO 字段、状态落点以及是否需要新增契约。baseline 明确不落状态时，本阶段不得建议依赖项目落状态。

不适用跨项目依赖时在对应表中明确写“不涉及”及原因。

### 7. 维护唯一 Claim Ledger

所有影响技术方案、SQL、接口、任务拆分或复用边界的关键判断，都只在 Claim Ledger 维护证据覆盖度、置信度和运行时缺口。Baseline 验证清单和第 3-7 节明细表只描述验证结果并引用 `Cxx`，不再复制一张覆盖度表或重复解释同一结论。

规则：

- 每个 `Cxx` 至少引用一个存在于代码证据索引的 `Exx`。
- 共享状态、枚举和类型语义矩阵中的每个真实消费场景必须引用类型为“共享语义影响”的 `Cxx`。
- `Bxx`、`Cxx`、`Exx`、`Qxx` 各自唯一，不得用重复编号表达不同事实。
- 文件类 `Exx` 写真实 `path:line`；运行时、接口、日志、DB/ES/Redis 证据明确写证据类型和来源。
- 证据等级只允许：代码已证实、编译已证实、接口已证实、数据已证实、推断、未覆盖、阻塞。
- 置信度只允许：高、中、低。存在未覆盖范围或运行时证据缺口时不能标高。
- 运行时证据缺口只允许：配置确认、接口验证、日志运行时确认、测试数据确认、用户口径确认、无。
- 推断、未覆盖必须写清真实范围和后续确认方式，禁止填写“无 / 不涉及 / 无需确认”，也不能作为确定方案依据。
- Baseline 为“已验证”时，关联 Cxx 必须已证实且没有未覆盖范围或运行时缺口；Baseline 为“未覆盖”时，关联 Cxx 不得写成已证实或高置信。
- 只有代码、编译、接口或数据已证实、置信度为“高”，且未覆盖范围和运行时证据缺口都为“无”的 Cxx，才能作为技术方案的确定依据。
- 阻塞结论必须在未覆盖范围或后续确认方式中引用 `Qxx`，且不能完成本阶段。

### 8. 维护唯一疑问账本并清零

`01-research.md` 的 Qxx 疑问账本是本阶段唯一的疑问和阻塞来源，不创建或维护 `01-blocking-issues.md`，也不把阻塞数独立写入 `meta.json`。

只有三类问题进入 Qxx：

- `用户意图`：材料不能确定业务期望；由用户确认并回写 baseline。
- `代码事实`：证据冲突或缺口会影响方案判断；由需求对齐阶段继续查证，不能让用户替代码作证。标为“已确认”时必须引用一个已闭合的 Cxx。
- `设计选择`：业务和代码事实明确，仅实现方式未定；写明边界后转交技术方案。

普通非阻塞证据缺口只写对应 `Cxx` 的未覆盖范围或运行时证据缺口，不重复登记 Qxx。Baseline、Claim Ledger 或残余风险一旦写“阻塞”，必须引用 Qxx；只要任一处仍为阻塞，`validate` 和 `to-design` 都不得通过。

需要用户澄清时，先读取提问模板。每次只问一个可独立决定的变量，用户可见内容只有：

- **当前疑问**：必要上下文加直接问题。
- **推荐理解**：明确推荐；证据不足时写“暂无可靠推荐”。

来源、证据、不确定原因、影响范围和推荐依据只写账本。推荐不能代替用户确认，回答也不能外推到未询问场景。

### 9. 回写权威文档

- 改变范围、用户路径、业务规则、数据身份、旧链路隔离、禁止项或验收标准：回写 `00-baseline.md`，重开澄清门禁并重新确认。
- 只改变代码事实、复用边界、落点或依赖能力：更新 `01-research.md`。

删除或改写失效口径，禁止新旧结论并存。baseline 变化后执行：

```bash
python3 "${CODEX_HOME:-$HOME/.codex}/skills/ggg-workflow-shared/scripts/workflow_cli.py" \
  sync-clarification --feature-dir <需求目录> --impact baseline research \
  --source <用户消息或新证据> --summary <变化摘要>
```

重新完成 baseline 确认后，受影响的 `Cxx` 必须重新查证。

### 10. 校验并收口

```bash
python3 "${CODEX_HOME:-$HOME/.codex}/skills/ggg-workflow-shared/scripts/workflow_cli.py" \
  sync-meta --feature-dir <需求目录>
python3 "${CODEX_HOME:-$HOME/.codex}/skills/ggg-workflow-shared/scripts/workflow_cli.py" \
  validate --feature-dir <需求目录>
```

`validate` 是进入技术方案的统一门禁，必须同时保证：

- Qxx 没有待确认项；用户意图已确认、代码事实已查证、设计选择已正确转交。
- Baseline 验证清单没有“阻塞”。
- Claim Ledger 没有“阻塞”证据等级。
- 残余风险中的“阻塞风险”为“无”。
- 非阻塞未覆盖项均已落到对应 `Cxx`，有后续确认方式且不会被当作确定事实。
- Baseline 的全部 Bxx 已逐项验证；主链路和第 3～7 节没有空模板内容。
- 共享语义影响判断已填写；涉及状态、枚举、类型码、业务阶段、场景标识或字段语义变化时，检索范围和消费场景矩阵已闭合。
- Bxx、Cxx、Exx、Qxx 均无重复，Baseline 状态与关联 Claim 的证据等级、置信度和缺口一致。

## 硬约束

- 禁止本阶段生成 `02-design.md`、`03-tasks.md` 或 `04-schema.sql`。
- 禁止跳过 baseline 验证清单，或只验证单个接口而不沿用户闭环检查数据承载、异步处理和重试。
- 禁止在新增或修改共享状态、枚举、类型码、业务阶段、场景标识或字段语义时，只沿主入口调用链调研，或只搜索枚举类名后声称影响面已闭合。
- 禁止无边界全仓扫描，禁止用扫描候选替代调用链证据，禁止输出配置值。
- 禁止让用户判断代码相关性或替代码作证；业务意图不明时才问用户。
- 禁止一次问多个独立问题，禁止用推荐或代码现状代替用户确认。
- 禁止把“未找到”写成“没有”，把低置信推断写成事实，或让缺少 `Cxx/Exx` 的关键结论进入下游。
- 禁止遗漏、合并或虚构 Bxx，禁止用空模板、“无”或“无需确认”伪装调研闭环。
- 禁止独立维护覆盖度表、阻塞文档或 `blocking_issue_count`；实时状态只从唯一 Qxx 账本计算。
- 禁止带阻塞进入下一阶段；非阻塞残余风险可以保留，但必须分类、说明范围和后续确认方式。

## 完成标准

- `01-research.md` 能讲清主链路、数据身份、重试路径、旧链路副作用、复用边界和跨项目依赖。
- 共享语义影响判断已收口；涉及共享状态或枚举变化时，生产方、载体、传播路径和所有已发现消费场景均已进入影响矩阵并引用 Cxx/Exx。
- 每个关键判断进入唯一 Claim Ledger，并有可追溯的 `Cxx`、`Exx`、证据等级和置信度。
- Baseline 的全部 Bxx 逐项验证；主链路和第 3-7 节有真实内容或“`不涉及：具体原因`”。
- Baseline 验证清单和第 3-7 节明细表引用 `Cxx`，编号唯一，没有重复覆盖度或重复结论账本。
- Qxx 是唯一疑问和阻塞账本，所有问题按类型清零或正确转交。
- Baseline、Claim Ledger、残余风险均无阻塞信号。
- `sync-meta` 和 `validate` 通过，可以进入 `$ggg-technical-design`。
