# GGG Task Breakdown（任务拆分）

## 定位

方案确认了，GGG 把它拆成程序员可以直接执行、排期的任务。

本阶段不写代码、不改 Java、不补 Mapper、不改 SQL。只生成 `03-tasks.md`，把 `00-baseline.md`、`01-research.md`、`02-design.md`、`04-schema.sql` 和 `interface-details/` 里的设计落成任务、依赖、并行等级、并行安全判断和完成标准。

任务必须能向上追溯。每个任务都要填写 `来源依据`，至少引用一个 `Dxx` 设计决策、`Cxx` 代码结论、接口明细文档或 `04-schema.sql`，不能只凭“需要开发”拆任务。

并行组是排期和派工依据，不是默认自动开多路写代码的命令。并行能力按 L0/L1/L2 降噪：默认先判断等级，再决定输出物。真正进入编码实现时，只有用户明确要求多 agent 并行编码，且本阶段为 L2，`ggg-implementation` 才能按 worker 分配表启动多个 worker；启动前仍需按真实文件、契约和代码事实做二次冲突确认。

## 显式触发规则

- 只有用户明确指定 `$ggg-task-breakdown` 或明确要求"开始任务拆分"时才执行。
- 如果 `02-design.md` 还没确认或 validate 未通过，先列缺口。

共享资源：

- 脚本：`../ggg-workflow-shared/scripts/workflow_cli.py`
- 任务模板：`../ggg-workflow-shared/assets/workflow/templates/task-breakdown-template.md`

## 前置条件

- `02-design.md` 已完成且确认
- 如涉及 MySQL 变更，`04-schema.sql` 已获得用户确认
- `interface-details/` 已与技术方案一致
- validate 已通过
- 无新的阻塞问题

## 执行步骤

### 第一步：推进阶段

```bash
py ../ggg-workflow-shared/scripts/workflow_cli.py to-tasks --feature-dir <需求目录> --design-confirmed
```

### 第二步：读取输入并建立覆盖清单

拆任务前先读取并对照：

- `00-baseline.md`：本期范围、禁止项、验收标准。
- `01-research.md`：主链路、旧链路副作用、复用边界、残余风险。
- `02-design.md`：核心改动、数据承载、主链路、接口、配置、发布和风险。
- `04-schema.sql`：涉及 MySQL 时必须覆盖 SQL 执行、实体/Mapper、验证和回滚任务。
- `interface-details/`：每个接口明细至少映射到实现任务和测试任务。

形成覆盖清单：

```text
| 来源 | 条目 | 来源依据 | 对应任务 | 是否覆盖 |
|---|---|---|---|---|
```

来源依据规则：

- 来自 `01-research.md` 的规则、复用边界、旧链路风险，写 `Cxx`。
- 来自 `02-design.md` 的设计决策、核心改动、接口设计，写 `Dxx`，并保留对应 `Cxx`。
- 来自接口明细，写具体 `interface-details/02-interface-XX-xxx.md`。
- 来自 SQL，写 `04-schema.sql`，必要时补对应 `Dxx/Cxx`。

### 第三步：拆任务

基于方案拆分，推荐顺序：基础准备 → 数据/SQL → 契约/DTO → 业务实现 → 接口与联调 → 发布与回退。

每项任务写清：编号、阶段、类型、来源依据、所属项目、主要落点、预计写入文件或目录、依赖、可并行组、初始状态、输出物、完成标准、工作量、协作方。

任务初始状态只使用：

- `ready`：前置依赖已满足，可以直接开始。
- `blocked`：依赖 SQL、接口契约、外部项目、用户确认或环境条件，当前不能开始。
- `pending`：需要排在前置任务后，但不是外部阻塞。

并行组建议：

- `P0`：SQL / 配置 / 环境前置。
- `P1`：实体、Mapper、DTO、枚举、配置类等可并行基础改动。
- `P2`：核心 Service / Manager / Facade 编排。
- `P3`：Controller / Dubbo / HTTP 接口与接口契约对齐。
- `P4`：测试、回归、发布、回滚。

并行组不是固定模板；按真实依赖调整。任务不能只写“开发接口”“实现逻辑”“测试一下”，必须写清主要落点和完成标准。

### 第四步：标记依赖、关键路径和并行等级

必须输出：

- 串行关键路径：例如 `T1 -> T2 -> T5 -> T8`。
- 可并行任务组：说明哪些任务可以同时做，哪些不能。
- 并行执行等级：只允许 `L0 / L1 / L2`。
- 阻塞任务：说明阻塞原因和解除条件。
- SQL 依赖任务：涉及 DDL 的实体、Mapper、Repository、验证任务必须依赖 SQL 确认/执行。
- 跨项目依赖任务：reasearch-hub、poseidon、2b、jzx-server 等项目边界要清楚。

并行执行等级：

- `L0 不并行`：任务集中在同一 Service / Mapper / 接口契约 / 配置 / 事务链路，或任务太小不值得并行。只写不并行原因，不强制输出并行安全矩阵和 worker 表。
- `L1 并行可行性评估`：存在可拆分任务，但本次默认单 agent 或串行实现。必须输出并行安全评估矩阵，不输出 worker 分配表。
- `L2 多 agent 并行准备`：用户明确希望并行，或任务天然独立且值得并行。必须输出并行安全评估矩阵和多 agent worker 分配表。

L1 / L2 必须输出并行安全评估矩阵：

```text
| 任务 | 并行组 | 预计写入文件/目录 | 共享依赖 | 契约/表/配置影响 | 冲突结论 | 执行方式 |
|---|---|---|---|---|---|---|
```

冲突结论只使用：

- `可并行`：预计不写同一文件、不抢同一契约、不依赖未完成任务。
- `需串行`：同组任务存在同文件、同主链路、同 Mapper XML、同配置或同接口契约冲突。
- `先锁契约`：Provider/Consumer、前后端接口、MQ 生产/消费、SQL 字段或配置口径未固定。

执行方式只使用：

- `并行`：可交给不同执行者同时做。
- `串行`：必须按依赖或文件冲突顺序做。
- `准备并行`：可先准备测试、样例、数据或文档，但不能给出实现通过结论。

判定规则：

- 同一个 Service 主链路、同一个 Mapper XML、同一个配置文件、同一个接口契约不能并行改。
- Provider/Consumer、前后端接口、MQ 生产/消费两端必须先锁定契约。
- SQL 未确认或未执行前，依赖表结构的实体、Mapper、Repository、测试数据任务不能标为 `ready`。
- 同一个并行组内可以有串行任务；`P1/P2/P3` 只是阶段标签，不等于所有任务都能并行。
- 测试任务可以和实现任务准备并行，但最终验证必须等实现任务完成。

只有 L2 才必须输出多 agent 并行编码分配表：

```text
| Worker | 任务 | 允许修改文件/目录 | 禁止修改文件/目录 | 只读参考文件/目录 | 依赖任务 | 合并顺序 | 验证命令 |
|---|---|---|---|---|---|---|---|
```

分配规则：

- 只有冲突结论为 `可并行` 且执行方式为 `并行` 的任务，才能分配给不同 worker 同时写代码。
- 每个 worker 必须有明确的 `允许修改文件/目录`，优先写精确文件路径；目录级写权限只用于新增文件，且必须保留唯一子目录或命名前缀。
- validate 只能稳定拦截精确路径重复；目录包含关系、通配符范围和新增文件命名冲突必须在本阶段写清并由实现阶段 coordinator 二次确认。
- 同一个文件、Mapper XML、配置文件、接口契约文件、SQL 文件不能出现在两个 worker 的允许修改范围里。
- 如果两个任务都需要改同一个 Service、Mapper XML、配置或接口契约，必须放到同一个 worker 或改为串行。
- 新增文件也要提前预留路径或命名前缀，避免两个 worker 新增同名 DTO、枚举、Mapper 或测试类。
- `禁止修改文件/目录` 必须写清共享主链路、共享契约、SQL、配置和其他 worker 已锁定的文件。
- `只读参考文件/目录` 用于让 worker 查上下文，但不能修改。
- `合并顺序` 必须能串起来，例如 `W1 -> W2 -> W3`；基础层一般先于主链路，主链路先于入口，入口先于测试结论。
- 每个 worker 必须有自己的局部验证命令或验证方式。

L0 / L1 不得为了满足模板伪造 worker：

- L0：写清“不并行原因”和“推荐单 agent / 串行执行顺序”。
- L1：写清可并行性和冲突点，但不输出 worker 表。
- L2：必须输出 worker 表，且每个 worker 的允许修改范围不能重叠。

### 第五步：映射接口、风险和测试

- 每个接口明细至少映射到一个实现任务和一个测试任务。
- 每个旧链路副作用或残余风险至少映射到实现防护、回归验证或发布检查任务。
- 每个 SQL 变更至少映射到执行/验证/回滚任务。
- 每个配置、Redis、MQ、ES 变更至少映射到配置确认、运行时验证或回归任务。
- 测试任务不能只写“测试”，必须绑定接口、SQL、缓存/MQ/配置或回归场景。

### 第六步：校验

```bash
py ../ggg-workflow-shared/scripts/workflow_cli.py validate --feature-dir <需求目录>
```

## 用户纠正处理

如果用户在本阶段指出文档有误或补充了新信息，直接在当前阶段更新对应文档。

## 硬约束

- 方案未确认，禁止拆任务
- 本阶段禁止写代码、改 SQL、改配置或生成实现文件
- 不新增方案里没有的需求
- 不把"待讨论"伪装成可开发任务
- 不允许只写"开发接口""联调一下"这类抽象任务
- 不允许任务缺少依赖、并行组、初始状态、输出物和完成标准
- 不允许任务缺少 `来源依据`；每个任务至少引用 `Dxx/Cxx/interface-details/04-schema.sql` 之一
- 不允许任务引用 `02-design.md` 中不存在的 `Dxx` 或 `01-research.md` 中不存在的 `Cxx`
- 不允许任务缺少预计写入文件或目录
- 不允许缺少并行执行等级 `L0 / L1 / L2`
- L0 不允许只写“不并行”但不说明原因和推荐串行顺序
- L1 / L2 不允许只写并行组，不写并行安全评估矩阵
- L0 / L1 不允许伪造 worker 分配表；L2 不允许缺少多 agent 并行编码分配表
- 不允许把同文件、同接口契约、同 Mapper XML、同配置项任务标为可并行
- L2 不允许两个 worker 的允许修改文件/目录存在同一文件、同一 Mapper XML、同一配置文件、同一接口契约或同一 SQL 文件
- L2 不允许 worker 缺少禁止修改文件/目录、合并顺序或验证命令
- 不允许接口明细、SQL 变更、旧链路风险没有对应任务
- 不允许把 blocked 任务标为 ready

## 完成标准

- `03-tasks.md` 可直接用于排期或派工
- 每项任务有明确归属、依赖、并行组、初始状态、输出物和完成标准
- 每项任务有明确来源依据，可追溯到 `Dxx/Cxx/interface-details/04-schema.sql`
- 已输出关键路径、可并行组和并行执行等级；L0 已说明不并行原因，L1 已输出并行安全评估矩阵，L2 已输出并行安全评估矩阵和多 agent 并行编码分配表
- 已覆盖 `02-design.md`、`04-schema.sql`、`interface-details/`、风险和测试链路
- 已覆盖发布和回退
- validate 通过
