# GGG Task Breakdown（任务拆分）

## 定位

把已确认的技术方案翻译成程序员可以直接执行的 `03-tasks.md`。本阶段只规划仓库内的编码工作，不写代码、不修改 SQL 或配置。

任务拆分回答四个问题：

1. 要实现哪一块代码能力。
2. 预计修改哪个项目、目录、类或方法。
3. 与其他编码任务有什么依赖。
4. 代码达到什么结果才算完成。

共享资源：

- 脚本：`../ggg-workflow-shared/scripts/workflow_cli.py`
- 模板：`../ggg-workflow-shared/assets/workflow/templates/task-breakdown-template.md`

## 前置条件

- `02-design.md` 已完成、已确认且 validate 通过。
- 存在 MySQL 变更时，`04-schema.sql` 已确认。
- `interface-details/` 与方案一致。
- 业务目标、接口契约、数据模型或技术归属仍有疑问时，停止拆分并回到对应阶段澄清；禁止把疑问包装成“前置任务”。

## 范围边界

只拆会产生仓库改动的编码任务，包括 Java、前端、SQL 文件、配置文件、脚本和实现风险要求的测试代码。

以下内容不生成任务：

- 接口命名确认、方案评审或契约锁定；它们是进入本阶段前必须完成的条件。
- 人工执行 DDL、数据回填、Nacos 配置、环境准备或联调协调。
- 发布顺序、上线检查、灰度、监控、回滚和发布单。
- 接口测试、回归测试或测试报告执行；这些由 `$ggg-test-verify` 负责。

状态机、事务、权限、公共契约、SQL、异步副作用和 Bug 回归所需的单元测试、集成测试、契约测试、测试桩或数据构造，直接并入对应功能 Txx，无需用户额外授权。默认不单独拆“测试代码”任务；只有测试资产本身是可被多个功能复用的独立代码能力时，才可单独拆成 Txx。无论是否独立，测试执行和测试报告都不生成编码任务。

## 执行步骤

### 第一步：推进阶段并读取输入

```bash
python3 "${CODEX_HOME:-$HOME/.codex}/skills/ggg-workflow-shared/scripts/workflow_cli.py" to-tasks --feature-dir <需求目录> --design-confirmed
```

逐项读取 `00-baseline.md`、`01-research.md`、`02-design.md`，以及按需存在的 `04-schema.sql` 和 `interface-details/`。只提取需要落到仓库中的编码改动：项目、代码职责、接口、数据读写、状态流转、异常边界、兼容要求，以及上述高风险行为所需的测试代码。测试执行步骤和测试报告只作为完成标准的验证输入，不转换成任务。

代码落点或依赖关系不确定时，查看相关真实代码再写任务；不要猜测文件路径、类名或复用关系，也不要让用户判断技术相关性。

### 第二步：先形成代码改动单元

以“可独立实现、可独立检查的一块代码能力”为一个候选单元。优先围绕业务能力或完整调用链拆分，不按 Controller、Service、Mapper 等分层机械拆分。

出现以下情况时通常应拆开：

- 属于不同 Git 项目或不同 Provider/Consumer。
- 有明确的先后依赖，例如 Provider 能力先于 Consumer 接入。
- 是可以独立完成和检查的持久化能力、核心业务能力或独立接口能力。
- 一个候选任务同时包含多条无直接关系的业务链路、状态机或数据模型。

出现以下情况时通常应合并：

- DTO、枚举、Controller、Service、Mapper 共同完成同一个小型业务行为。
- SQL 文件、Entity、Mapper/Repository 共同承载同一份简单数据能力。
- 配置文件或配置读取代码只服务于某个功能，没有独立交付价值。
- 状态机、事务、权限、公共契约、SQL、异步副作用或 Bug 回归所需测试代码只验证对应功能行为。
- 单独拆出后只能写成“新增一个类”“补注释”“改格式”等没有完整结果的任务。

只有测试基座、共享 Fixture、通用数据工厂、契约测试工具等测试资产本身形成可独立复用的代码能力，且主要实现内容或代码边界明确至少两个具体复用方或跨模块复用范围时，才考虑单独拆任务。裸写“可复用”“共享”或把复用口号写进完成标准都不能作为单拆依据。

任务数量由真实代码边界决定，不为了显得完整而凑任务，也不把一个大功能笼统写成一个任务。

### 第三步：确定依赖

只记录编码任务之间由真实代码和契约产生的依赖。

- 没有依赖写 `-`。
- 依赖只能引用当前文档中存在的 `Txx`，不能自依赖或循环依赖。
- 不允许把人工 SQL、环境配置、联调、测试或发布作为编码依赖。
- 若编码仍依赖未确认的业务或技术事实，停止生成正式任务并回写上游文档。

### 第四步：写任务

每个任务必须包含：

- `编号`：唯一的 `Txx`。
- `开发任务`：使用“实现/新增/调整 + 代码能力”的具体名称。
- `来源依据`：必须至少引用一个 `02-design.md`“核心改动”中的真实 `Dxx`；`Cxx`、接口明细和 `04-schema.sql` 只能作为补充依据，不能单独产生编码任务。
- `所属项目`：实际写入的 Git 项目；跨项目必须拆任务。
- `预计修改文件/符号`：基于代码事实填写目录、类、方法或 SQL 文件，不确定时先查代码。
- `依赖任务`：只写任务编号或 `-`。
- `完成标准`：以直接子项分别写 `代码结果与关键行为`、`必要测试代码` 和 `最小验证`，每项只出现一次且有实质内容；测试不涉及时写具体原因，高风险行为不得用“不涉及”豁免。最小验证必须落到具体命令、测试类/用例、文件/接口或具名静态检查，不能只写“执行所有测试”“验证功能行为”。

任务总览只维护编号、开发任务、所属项目和依赖任务。任务详情唯一维护来源依据、预计修改文件/符号、主要实现内容、代码边界和完成标准，不重复所属项目或依赖任务。不要粘贴完整代码，不重复技术方案，也不要写工作量、协作方、发布风险、上线方式或人工操作清单。

代码规范、必要注释/Javadoc、日志、异常处理、参数校验和格式要求是所有任务的共同完成要求，不单独拆任务；实现阶段继续加载相应代码规范 Skill。

### 第五步：自检和校验

生成前逐项确认：

- `02-design.md`“核心改动”中的每个 `Dxx` 都至少被一个合理的编码任务承载；任务不得只引用 `Cxx` 或其他辅助材料。
- 每份接口明细都出现在某个编码任务的来源或实现内容中。
- 存在 `04-schema.sql` 时，至少一个编码任务覆盖 SQL 文件及对应持久化代码。
- 没有遗漏跨项目 Provider/Consumer，也没有把同一代码改动重复分配给多个任务。
- 总览与详情任务一致，编号唯一，依赖存在且无环。
- 没有发布、回滚、人工 DDL、环境操作或测试执行任务。
- 状态机、事务、权限、公共契约、SQL、异步副作用和 Bug 回归所需测试代码已并入对应功能 Txx。
- 没有默认拆出独立测试任务；独立测试任务写明了测试资产类型和至少两个具体复用方或跨模块复用范围。
- 每个 Txx 的完成标准都有唯一、非空的 `必要测试代码` 和 `最小验证` 直接子项，最小验证与风险相称，不把测试执行过程拆成任务。

```bash
python3 "${CODEX_HOME:-$HOME/.codex}/skills/ggg-workflow-shared/scripts/workflow_cli.py" validate --feature-dir <需求目录>
```

validate 通过后按既有授权继续：

- 用户已经授权连续编码，且任务拆分没有新增需要用户决定的业务、成本、兼容或高风险取舍时，直接执行以下交接命令，不再增加一次形式化确认：

```bash
python3 "${CODEX_HOME:-$HOME/.codex}/skills/ggg-workflow-shared/scripts/workflow_cli.py" to-implementation \
  --feature-dir <需求目录> \
  --tasks-confirmed
```

- 用户尚未授权编码时，只给出精简任务摘要并等待明确确认；确认前不执行 `to-implementation`，也不启动 implementation。

## 用户纠正处理

只影响任务表达时更新 `03-tasks.md`；影响方案时执行 `sync-clarification --impact design tasks`；影响需求基线时执行 `sync-clarification --impact baseline research design tasks`，重新澄清并执行 `confirm-baseline`。禁止绕过上游权威文档直接改变代码范围。

## 完成标准

- `03-tasks.md` 只包含可产生仓库改动的编码任务。
- 程序员可以仅根据方案和任务清单确定先改哪个项目、哪些代码以及完成边界。
- 任务粒度不过度分层，也没有把多条无关链路塞进同一任务。
- 来源、项目、代码落点、依赖和完成标准真实、完整且一致。
- 高风险行为和 Bug 回归所需测试代码已纳入相应功能任务，完成标准包含最小验证。
- validate 通过。
