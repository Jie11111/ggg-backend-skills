# GGG Technical Design（技术方案）

## 定位

GGG 就是那个程序员，需求和代码都确认了，开始写技术方案。

**身份与契约骨架先行，数据承载随后，SQL 经用户确认后再写完整方案**。

**设计必须追溯到代码结论**。凡是会影响 SQL、核心改动、接口契约、配置、缓存、MQ、ES、任务拆分或发布风险的设计项，都必须引用 `01-research.md` 的 `Cxx` 结论；设计决策使用 `D1/D2/...` 编号，供 `03-tasks.md` 继续追溯。

不要一上来拍 SQL。先确认这条业务记录唯一由什么标识、状态按什么维度隔离、前端页面动作会触发哪些后端接口、哪些关键参数必须由后端推导，再决定 MySQL / Redis / ES / MQ / 配置分别承载什么。

这里的"前后端接口协作流"不是前端开发方案，只是后端接口契约视角：页面动作、接口粒度、首屏轻重、点击后查询、后端自动推导、前端禁止传参。

SQL 存储层是技术方案的硬门禁。凡是涉及 MySQL 表结构新增或修改，必须先单独产出 `04-schema.sql` 并让用户确认；用户确认 SQL 没问题后，才真正进入 `02-design.md` 主体和 `interface-details/` 设计。

## 显式触发规则

- 只有用户明确指定 `$ggg-technical-design` 或明确要求"开始技术方案 / 写方案"时才执行。
- 如果需求对齐未完成或 validate 未通过，先列缺口，不开始写方案。

## 执行步骤

### 第一步：推进阶段

```bash
py ../ggg-workflow-shared/scripts/workflow_cli.py to-design --feature-dir <需求目录>
```

### 第二步：设计预检

先读取 `00-baseline.md` 和 `01-research.md`，完成三张轻量表。三张表没有收口前，不要写 `04-schema.sql`，也不要拆接口明细。

#### 2.1 实例身份确认

先回答"这条业务记录唯一由什么标识"。

建议格式：

```text
| 业务对象/记录 | 唯一标识 | 状态隔离维度 | 去重维度 | 生命周期 | 来源证据 | 是否已确认 |
|---|---|---|---|---|---|---|
```

必须覆盖：

- 主记录、明细记录、队列记录、进度记录、关系记录。
- 前端可能传入但后端不应信任的身份字段。
- 后端必须自动推导的身份字段。
- 状态隔离维度，例如按 `selectionId`、`modulePtId`、`ptQueueId`、`userId + bizId`。

#### 2.2 前后端接口协作流骨架

只写页面动作到后端契约，不写前端实现细节。

建议格式：

```text
| 页面/动作 | 调用接口 | 首屏/点击后 | 请求关键字段 | 后端自动推导 | 前端禁止传 | 返回粒度 | 说明 |
|---|---|---|---|---|---|---|---|
```

重点确认：

- 首屏接口轻重：首屏只返回必要数据，重数据点击后再查。
- 后端自动推导：例如从 active selection 推导当前实例，不让前端传错关键身份。
- 前端禁止传参：凡是会污染实例身份、状态隔离、权限范围的字段，都不要由前端传。
- 查询和写入分离：预览、点击、开始训练、提交反馈等动作不要混成一个大接口。

#### 2.3 数据承载选择

所有存储层一起考虑，不只 SQL。

建议格式：

```text
| 数据/状态 | 承载方式 | MySQL | Redis | ES | MQ | 配置/缓存 | 选择原因 | 一致性/过期策略 |
|---|---|---|---|---|---|---|---|---|
```

选择原则：

- MySQL：长期业务状态、关系、审计、可回溯数据。
- Redis：短期状态、锁、热点缓存、临时进度；必须写清过期和一致性策略。
- ES：搜索、筛选、聚合，不承载唯一业务真相。
- MQ：异步事件、补偿、跨系统同步；必须写清幂等和失败处理。
- 配置/本地缓存：开关、阈值、灰度；必须写清刷新和默认值。

### 第三步：设计数据存储

实例身份、接口协作流骨架和数据承载选择确认后，再设计存储：

- SQL 字段风格参考：先找同模块 / 同子项目已有表，明确公共字段、逻辑删除字段、时间字段、用户字段和索引命名参考哪几张表。
- SQL 瘦身检查：先证明每张表和每个关键字段必须落库，再写 DDL。
- 需要哪些表（新增 / 修改）
- 关键字段、主键、唯一键、索引
- Redis Key、过期时间、一致性策略
- ES 索引 / 字段 / 同步来源
- MQ Topic、消息体、幂等键、重试策略
- 配置项、默认值、灰度和回滚方式
- 新增表的 ER 图
- 历史数据处理

公共字段和 `user_id` 规则：

- 公共字段优先沿用目标模块已有表风格；找不到参考时，默认使用 `id`、`create_user`、`create_time`、`modify_user`、`modify_time`、`is_delete`。
- 如果目标模块已有表使用 `is_deleted`、`creator`、`update_time`、`deleted` 等历史字段，以目标模块为准，并在 SQL 说明里写明参考表。
- `user_id` 不是公共字段，不要默认加。只有业务记录确实归属某个用户、查询/权限/状态隔离需要按用户维度过滤，才设计 `user_id`。
- 设计 `user_id` 时必须说明：用户身份来源、是否前端可传、是否后端推导、是否参与唯一键/普通索引、和实例身份字段的关系。
- 如果存在学生、老师、创建人、操作者等多种用户语义，不要都叫 `user_id`；按业务语义命名，例如 `student_id`、`teacher_id`、`operator_id`、`create_user`。

SQL 瘦身检查：

```text
| 表 | 业务事实 | 写入事件 | 查询场景 | 生命周期 | 能否复用旧表 | 不建表后果 |
|---|---|---|---|---|---|---|
| 字段 | 来源 | 写入时机 | 读取场景 | 是否可推导 | 是否参与查询/唯一键/索引 | 不落库后果 |
```

瘦身规则：

- 只为"以后可能用"设计的表或字段，不落库。
- 能从现有表、上游实时计算、关联表稳定推导的字段，不默认落库。
- 只为排查方便保留的冗余字段，必须写清冗余原因和不一致风险。
- 计数、布尔状态、最近状态不要重复表达；例如能由 `count > 0` 推导时，不默认再存 `learned`。
- 新增表优先承载一个明确业务事实，避免把展示快照、统计字段、过程日志混在一张表。
- 多张表之间如果生命周期、写入事件和查询场景一致，必须说明为什么不能合并。

如涉及 MySQL 变更，写入 `04-schema.sql`，先和用户确认表设计没问题。

```bash
py ../ggg-workflow-shared/scripts/workflow_cli.py to-design --feature-dir <需求目录> --create-schema --business-model-confirmed --upstream-contract-confirmed
```

如果本需求不涉及 MySQL 变更，不要创建 `04-schema.sql`，必须在 `02-design.md` 的数据承载选择中明确说明"无 MySQL 结构变更"以及原因。

SQL 确认门禁：

- 涉及 MySQL 变更时，本步骤只推进到 `04-schema.sql` 评审，不要继续写 `02-design.md` 主体和 `interface-details/`。
- 向用户说明表、字段、主键、唯一键、索引、历史数据处理和兼容影响。
- 用户明确确认 SQL 没问题后，才进入第四步。
- 如果用户调整 SQL 口径，先修改 `04-schema.sql`，必要时回写实例身份、数据承载选择，再重新确认。
- 不涉及 MySQL 变更时，说明原因后可以直接进入第四步。

### 第四步：写技术方案主体

设计预检和数据存储确认后写 `02-design.md`。如果涉及 MySQL 变更，必须等用户确认 `04-schema.sql` 后再写主体；如果不涉及 MySQL 变更，必须在数据承载设计里说明原因后再写主体。按模板章节顺序：

1. **背景与目标**——改什么、为什么改
2. **实例身份与状态隔离**——业务记录唯一标识、状态隔离、去重维度、生命周期
3. **前后端接口协作流**——页面动作到后端接口契约，不写前端实现
4. **数据承载设计**——MySQL / Redis / ES / MQ / 配置分别承载什么
5. **SQL 表设计**——表结构概览、字段设计、ER 图（引用 04-schema.sql）
6. **核心改动**——改哪些类、方法、模块
7. **主链路与依赖**——系统交互主时序图、调用链、Dubbo/HTTP 依赖
8. **接口设计**——接口总表 + 各接口详情文档
9. **枚举/状态/常量**
10. **影响范围**
11. **测试链路与风险**

其中下面章节必须显式引用 `Cxx`：

- **SQL 表设计**：每张新增/修改表、关键字段或“无 MySQL 结构变更”结论必须引用支撑它的 `Cxx`。
- **核心改动**：每条 Controller / Service / Facade / DTO / Mapper / SQL / 配置改动必须引用 `Cxx`。
- **接口设计**：每个新增/修改接口必须引用 `Cxx`，接口明细要继承同一个来源口径。
- **设计决策记录**：每条 `Dxx` 决策必须引用至少一个 `Cxx`，不能凭方案偏好新增设计。

### 第五步：拆接口明细

每个接口一份文档，放在 `interface-details/` 下，命名为 `02-interface-XX-接口主题.md`。

接口文档至少包含：
- 基本信息、请求/响应参数
- JSON 示例
- 处理流程图
- 异常处理
- 测试链路

接口明细必须落实协作流骨架：

- 请求参数里标注哪些来自前端，哪些由后端推导。
- 对前端禁止传的字段，接口文档必须说明后端如何获取。
- 响应字段按首屏 / 点击后 / 提交后区分粒度。
- 写接口必须说明幂等、重复提交、状态隔离和权限校验。

### 第六步：设计决策和残留口径自检

写一个轻量设计决策表，不需要完整 ADR：

```text
| 设计ID | 决策点 | 选择 | 不选方案 | 来源Cxx | 原因 | 影响 |
|---|---|---|---|---|---|---|
```

决策 ID 规则：

- 使用 `D1/D2/...`，每个会影响实现的关键设计点都要有 `Dxx`。
- `来源Cxx` 必须引用 `01-research.md` 结论账本中存在的结论；多个结论用 `C1、C2`。
- 如果某个设计只能来自用户口径而不是代码结论，要先回写 `00-baseline.md` 和 `01-research.md`，形成对应 `Cxx` 后再写入方案。
- 不允许 `Dxx` 的来源只写“需求要求 / 方案判断 / 经验做法”。

进入任务拆分前，手动检查 `02-design.md`、`04-schema.sql`、`interface-details/`：

- 不得保留 `TODO`、`TBD`、`FIXME`、`待补充`、`暂定`。
- 不得保留已经废弃的旧配置名、旧接口口径、旧字段语义。
- `待确认`、`后续确认` 只能出现在 `02-design.md` 的"测试链路与风险 / 关键风险 / 非阻塞风险"中，并写清为什么不影响进入任务拆分。
- `04-schema.sql` 和 `interface-details/` 不允许保留 `待确认`、`后续确认`。
- 阻塞问题必须回到上游文档或继续澄清，不能进入任务拆分。

py ../ggg-workflow-shared/scripts/workflow_cli.py sync-meta --feature-dir <需求目录>
py ../ggg-workflow-shared/scripts/workflow_cli.py validate --feature-dir <需求目录>

如果用户在本阶段指出文档有误或补充了新信息：

- 影响代码结论的 → 更新 `01-research.md`

## 写作要求

- 先把实例身份、接口协作流骨架、数据承载选择定清，再写存储和完整方案
- 设计项必须能追溯到 `Cxx`；新增 `Dxx` 决策后，下游任务必须能引用该 `Dxx`
- SQL 设计前必须列出参考表和字段风格，不直接套通用模板
- SQL 设计前必须完成表准入和字段准入检查，先删掉不必要的表和字段
- `user_id` 必须有明确业务语义、来源和查询/隔离依据，不作为默认公共字段
- 不要写完整前端方案；只写后端需要承诺和防错的接口契约
- 尽量用仓库中真实的类名、Facade 名、表名，不要发明新名词
- 主时序图画“系统交互”，不要画成类/方法调用细节图
- 接口风格沿用项目现有代码风格
- 不确定的地方问用户，不要自己猜了写进方案

### 主时序图要求

- 主时序图的参与方优先使用系统边界和基础设施：
  - 前端 / 调用方
  - 当前后端项目，例如 `2b/back`
  - 依赖项目或 Dubbo 服务，例如 `QuestionFacade`、`LearningTrackFacade`
  - MySQL / Redis / ES / MQ / WebSocket / 第三方服务
- 主时序图只表达“谁调用谁、传什么关键业务对象、返回什么关键结果”，不要展开 Controller、Service、Converter、Repository、Mapper 等内部类。
- 不要在主时序图里塞太多内部异常分支；只保留会影响前后端协作、数据一致性、异步消息或外部依赖的关键分支。普通参数校验、私有方法、DTO 转换、日志、trace 不进主时序图。
- 如果确实需要说明后端内部处理细节，放到“核心改动”“调用链”“接口明细流程图”里，不放进主时序图。
- PlantUML 写法要可渲染：
  - 带空格、斜杠、连字符或换行的参与方名称必须加引号，例如 `queue "learning-track-topic" as MQ`。
  - 一张主图优先控制在 6-10 个参与方以内；超过后先合并成项目/中间件级参与方。

## 硬约束

- 未完成需求对齐，禁止写方案
- 实例身份未确认，禁止设计数据存储
- 前后端接口协作流骨架未确认，禁止拆接口明细
- 数据承载选择未确认，禁止只写 SQL 或只写接口
- 涉及 MySQL 变更时，用户未确认 `04-schema.sql` 前，禁止写 `02-design.md` 主体和 `interface-details/`
- 未说明 SQL 参考表和公共字段风格，禁止提交 `04-schema.sql`
- 未完成表准入和字段准入检查，禁止提交 `04-schema.sql`
- 不允许为"以后可能用"、"排查方便"直接新增字段；必须说明读取场景、不落库后果和冗余风险
- 不允许无业务依据地默认添加 `user_id`
- **不确定就问**：方案中拿不准的设计决策，先和用户确认
- 禁止生成 `03-tasks.md`
- 不在方案里留未收口的阻塞问题
- 不在方案、SQL、接口明细里保留 `TODO`、`TBD`、`FIXME`、`待补充`、`暂定`、旧配置名、旧接口口径
- `待确认`、`后续确认` 只允许作为非阻塞风险写在 `02-design.md` 风险章节；不得出现在 SQL 或接口明细中
- 不把前端应禁止传的关键身份字段设计成普通请求参数
- 不用抽象词当方案（如"优化性能"但没有落点）
- 不允许 SQL 表设计、核心改动、接口设计或设计决策缺少 `Cxx` 来源
- 不允许 `Dxx` 设计决策引用 `01-research.md` 中不存在的 `Cxx`

## 完成标准

- 已确认实例身份、状态隔离维度、去重维度和生命周期
- 已产出前后端接口协作流骨架，并说明后端自动推导和前端禁止传的字段
- 已说明 MySQL / Redis / ES / MQ / 配置的数据承载选择；不涉及的已说明原因
- 已说明 SQL 参考表、公共字段风格，以及 `user_id` 是否需要、语义来源和索引依据
- 已完成表准入和字段准入检查，删除或说明了可推导、重复表达、仅未来可能使用的表和字段
- 如涉及 MySQL 变更，`04-schema.sql` 已写清表结构变更并已获得用户确认；不涉及则不创建 `04-schema.sql`，并已在数据承载设计中说明原因
- `02-design.md` 已形成可评审的技术方案
- 每个接口已有独立文档
- 已记录关键设计决策，每条 `Dxx` 都引用了有效 `Cxx`
- SQL 表设计、核心改动、接口设计均已引用 `Cxx`，下游任务可追溯
- 已完成残留口径自检
- 已包含主时序图、测试链路、关键风险
- validate 通过
- 可以进入任务拆分
