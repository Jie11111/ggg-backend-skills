# SQL 设计检查

仅在新增或修改 MySQL 结构时读取。新版流程先在需求对齐阶段把精确 SQL
写入 `sql-draft.sql` 并完成 SQL Gate，再进入技术方案；技术方案只引用已确认
SQL。历史需求仍可继续用 `04-schema.sql`，不自动迁移。

## 内容边界

`02-design.md` 记录：

- 为什么需要变更、复用过哪些现有结构。
- 表承载的业务事实、核心写入和核心查询。
- 索引或约束依据、历史数据、兼容顺序、回滚和验证。
- 每个 DDL 对象的操作、字段/索引/约束覆盖、风险等级与具体依据、对应 Cxx 和 Dxx。

新版 `sql-draft.sql`（历史流程为 `04-schema.sql`）唯一记录：

- 完整可执行 DDL。
- 字段类型、长度、空值、默认值和注释。
- 主键、唯一键、普通索引、表选项及必要验证 SQL。
- 每条结构 DDL 紧邻的一条 `GGG_DDL_OBJECT` JSON；`object`、`operation`、`members`、`risk`、`risk_reason`、`claims`、`designs` 必须与真实语句和主方案逐对象一致。

不要在两份文件中重复维护精确字段定义。

## 最小化检查

按“不改库 → 复用旧表 → 修改旧表 → 新建表”的顺序判断：

- 每张表只承载已确认的业务事实，不为未来可能使用而建表。
- 可由权威来源稳定推导的字段默认不落库；冗余字段必须说明同步和不一致风险。
- 状态、布尔值和计数不重复表达同一事实。
- `user_id` 不是默认公共字段；按学生、老师、操作者、创建人等真实语义命名，并说明身份来源和隔离/查询用途。
- 普通索引必须对应真实 `WHERE/JOIN/ORDER BY`，先检查最左前缀和重复索引。
- 唯一键只表达真实业务唯一性，不用普通索引代替。

## DDL 风险判断

出现任一情况即按高风险 DDL 处理：

- 删除、重命名或改变存量表/字段语义。
- 修改字段类型、长度、字符集、空值、默认值、主键或唯一约束。
- 批量回填、数据重写、双写、迁移或不可逆操作。
- 大表或热点表 ALTER、建索引，或者数据规模、MySQL 版本、在线 DDL 能力不清楚。
- 新旧代码无法兼容发布，或变更影响多个服务的读写顺序。
- 失败后不能通过停用新代码和回滚增量结构安全恢复。

普通 DDL 仅包括证据充分的低风险增量，例如隔离的新表、向后兼容的可空字段，或已验证规模和执行方式的非唯一索引。证据不足时按高风险处理。

## 确认时机

- 普通和高风险 DDL 都必须先在 SQL Gate 展示并获得用户确认，再创建或填写正式技术方案。
- SQL Gate 至少展示精确 DDL、结构、数据规模、锁/回填风险、兼容顺序、回滚边界和验证方式；风险不明时先补证据。
- 任何已确认 DDL 的表、字段、索引、约束、执行顺序或风险语义发生变化，都先使下游结论失效，再重新确认 SQL。新版使用 `confirm-sql`，历史需求继续使用 `confirm-schema`。
- 仅注释、空白和不改变语义的排版调整不要求重新确认。

## 提交前检查

- 参考目标模块真实表风格，不套通用字段模板。
- 核心查询与索引顺序相符；高风险查询按需执行 `EXPLAIN`。
- 写清历史数据处理、新旧代码兼容顺序、备份/回滚和验证 SQL。
- 不声称“无锁表风险”或“无影响”，除非有版本、规模和执行方式证据。
- 一条 DROP/RENAME TABLE 只处理一个对象；多对象操作拆成独立语句并逐条标记。
