# GGG Test Verify（测试验证）

## 定位

GGG 在这个阶段负责证明代码能跑、接口能通、主链路符合需求。它接在 `$ggg-code-review` 后面，不再做代码风格评审，也不默认改业务代码。

**用运行结果说话**：优先复用需求目录里的测试脚本、联调说明和历史报告；没有现成脚本时，再按接口文档和 Controller 生成可复用测试报告。

**把该测的都测掉**：本阶段不追求覆盖率数字，而是先列出应测场景清单，再逐项给出 `已验证 / 未验证 / 阻塞 / 不适用`、证据和原因。主链路、接口契约、SQL、权限、状态、异步和旧链路回归这类关键场景不能用“已跑部分接口”代替。

**最小必做门禁**：

1. 先列应测场景清单，至少覆盖本次主链路、接口契约变化和 review 指出的验证缺口。
2. 每个未测场景必须说明原因；关键场景未测或失败时，结论不能写 `通过`。
3. 所有 JSON 入参必须确认实际发送内容合法、`Content-Type` 正确，并与 `@RequestBody` DTO 字段类型一致。

其他权限、状态、幂等、SQL、Redis、MQ、ES、WebSocket、旧链路回归等场景，只在本次需求、设计、代码改动或 review 风险涉及时强制纳入应测清单。

**测试报告要按轮次资产化**：如果当前任务有需求目录，每次测试验证都必须维护 `07-test-report.md` 索引，并新增一份 `test-rounds/test-rNN.md` 明细。补测或修复后复测时新增下一轮，不覆盖上一轮；接口专项报告仍可继续落在 `reports/api-tests/`。

如果本次是无需求目录的 quick 小需求，不强制创建 `07-test-report.md` 和 `test-rounds/`；优先读取 `quick.md`，按 quick 边界、diff、review 缺口和代码事实建立轻量应测场景清单，并在最终回复给出验证结论，同时把验证记录回写 `quick.md`。只有用户要求报告、接口验证复杂、或存在未关闭风险时，才生成普通 Markdown 报告。

## 显式触发规则

- 只有用户明确指定 `$ggg-test-verify`，或明确要求“开始测试验证 / 跑接口测试 / 验证本次改动 / 生成测试报告 / 用 token 测接口链路 / quick 改动验证”时才执行。
- 如果用户要求检查代码质量，使用 `$ggg-code-review`。
- 如果用户要求继续实现或修复代码，使用 `$ggg-implementation`。
- 如果用户要求测试单个 JZX 后端接口、Controller 方法或 token 行为，可按需联动 `jzx-api-test`。

## 前置输入

优先读取：

- 当前需求目录：`00-baseline.md`、`01-research.md`。
- 当前需求目录：`02-design.md`、`03-tasks.md`、`04-schema.sql`、`interface-details/`、联调说明、`scripts/`、`reports/api-tests/`。
- `$ggg-code-review` 结论：尤其是验证缺口、风险点、并行合并审查结果、依赖/配置变更和未验证项。
- 当前需求目录里的 `06-code-review.md`、`review-rounds/`、`07-test-report.md` 和 `test-rounds/`，用于识别最新 review 结论、历史测试轮次和未关闭缺口。
- 本次相关 diff、代码落点、Controller、Service、Mapper、配置和测试文件。
- 用户提供或确认的环境、baseUrl、员工 token、学生 token、管理员 token 或登录账号。
- 项目已有编译、单测、接口测试命令。

如果是 quick 小需求且没有需求目录，优先读取 `ggg/quick/.../quick.md`，再读取用户确认的 quick 边界、实现说明、review 结论、本次 diff、相关代码落点和可用验证命令。

如果缺少 token 或环境地址，先从接口文档、历史脚本或 `jzx-api-test` 规则判断需要哪类 token，再向用户确认或索取；不要默认复用历史脚本里写死的 token。

## token 与安全规则

- 员工端、管理端、学生端 token 不能混用，必须按接口角色传。
- 执行需要登录态的接口测试前，必须让用户确认或提供本次使用的 token。
- 历史脚本里的 token 只能用来识别参数名和角色类型，不能默认当成本次可用 token。
- JZX 常见 HTTP Header 为 `ACCESS-TOKEN`；若当前项目有其他 Header，按代码事实处理。
- WebSocket 如果先通过 HTTP 获取 `wsToken`，握手时优先使用接口返回的 `wsPath` 或 token 参数。
- 最终报告和回复必须脱敏 token，只保留类似 `abcdef...1234` 的格式。
- 不在最终回复输出完整 token、Cookie、Authorization、账号密码。
- token 失效时，先说明失效现象和需要的角色 token；只有用户确认可刷新时，才使用已有刷新脚本或 `jzx-api-test` 约定刷新。

## 入参 JSON 规则

- 发送带 body 的 HTTP 请求前，必须先确认请求体是合法 JSON，字段名必须带双引号。
- Windows PowerShell 下不要手写 `curl.exe --data-raw '{"a":"b"}'` 这类 JSON；PowerShell 可能剥离引号，导致服务端收到 `{a:"b"}` 这类非法 JSON。
- 优先使用 `jzx-api-test`、已有测试脚本、BodyFile、PowerShell 原生对象转 JSON，或 .NET `HttpClient` 发送请求体，确保实际发送内容和报告里的 JSON 一致。
- Controller 方法存在 `@RequestBody` DTO 时，默认按 DTO 字段类型校验请求体；数字、字符串、数组、对象、枚举、布尔值不能随意互换。
- 常见类型不一致要在发送前修正，例如后端 DTO 是 `String`，测试 JSON 不要传裸数字；后端是 `List<Long>`，测试 JSON 不要传逗号字符串。
- `Content-Type` 必须按接口事实设置，JSON 请求使用 `application/json`；表单、文件上传、query 参数接口不能伪装成 JSON。
- 报告里必须记录实际发送的请求体；如果使用脚本或 BodyFile，要记录文件路径和脱敏后的请求 JSON。
- 如果出现 400、JSON parse error、`HttpMessageNotReadableException`、字段全为 null、枚举绑定失败，优先排查实际发送 JSON、Content-Type 和 DTO 类型映射，不要直接判定业务代码失败。

## 测试资产优先级

按下面顺序选测试方式：

1. 需求目录已有专项脚本：先阅读脚本默认参数、默认环境、默认 token 和写接口步骤，确认安全后优先执行，例如 `scripts/test_xxx_flow.py`。
2. 需求目录已有联调说明：按说明拆 HTTP、WebSocket、MQ、缓存、ES 验证点。
3. 需求目录已有历史报告：复用报告结构、步骤名、断言口径和输出格式。
4. 单接口或 Controller 方法：使用 `jzx-api-test` 生成 Markdown 报告。
5. 无现成资产：根据 `interface-details/` 和真实 Controller 手写最小测试步骤，并落地报告。

## 执行步骤

### 第一步：确认验证范围和应测场景

如果存在需求目录，且当前 `meta.json.current_phase` 仍是 `代码检查`，先确认最新 `$ggg-code-review` 结论为通过，再推进到测试验证阶段：

```bash
py ../ggg-workflow-shared/scripts/workflow_cli.py to-test --feature-dir <feature-dir> --review-passed
```

如果当前阶段已经是 `测试验证` 或 `交付完成`，不重复推进；直接按本轮验证继续。

先确认本次要验证什么：

- 编译是否通过。
- 单测是否通过。
- 哪些接口必须测。
- 是否涉及 WebSocket、MQ、Redis、ES、定时任务、事务后通知。
- 是否需要验证 SQL 结构、索引、字段映射或历史数据。
- 是否只验证本地环境，还是测试环境也要验证。

如果 `$ggg-code-review` 结论不是 `通过`，原则上先回到实现或评审阶段；用户明确要求带问题验证时，最终结论必须标记风险。

必须先建立应测场景清单：

```text
| 场景ID | 来源 | 应测场景 | 场景类型 | 验证方式 | 状态 | 证据 | 未测/阻塞原因 |
|---|---|---|---|---|---|---|---|
```

来源包括：

- `00-baseline.md`：验收标准、本期范围和禁止项。
- `01-research.md`：旧链路副作用、复用限制、残余风险。
- `02-design.md`：主链路、数据承载、状态、配置、发布和风险。
- `03-tasks.md`：每个开发任务、每个 worker 输出物、每个测试任务。
- `04-schema.sql`：表、字段、索引、默认值、回滚或历史数据处理。
- `interface-details/`：每个接口、入参、出参、异常、后端推导字段、前端禁止传字段。
- `$ggg-code-review`：必须补测项、验证缺口、安全/性能/兼容风险。
- git diff：实际改动文件、配置、依赖、Mapper、SQL、测试脚本。
- quick 边界：无需求目录时，优先用 `quick.md` 中的需求目标、改动范围、不做范围和验收信号作为来源。

场景类型建议：

- 编译 / 单测 / 接口 / 主链路 / 权限 / 参数校验 / 状态流转 / 幂等重复 / SQL / Redis / MQ / ES / WebSocket / 配置 / 旧链路回归 / 发布回滚。

状态只允许：

- `已验证`：实际执行并有命令、请求响应、日志、数据库回查或事件证据。
- `未验证`：当前没测，必须说明原因和影响。
- `阻塞`：执行失败或缺关键环境、token、数据、依赖，导致关键场景无法判断。
- `不适用`：需求明确不涉及，必须说明不适用原因。

不能用覆盖率百分比替代场景清单。

### 第二步：准备环境和 token

确认：

- `baseUrl`，例如 `http://127.0.0.1:16022` 或测试环境地址。
- 用户确认或提供的员工 / 管理员 / 学生 token。
- 需要的前置数据，例如课堂、学生、教室、口令、记录 ID。
- 服务是否已启动，端口是否可访问。
- 本次测试会产生或修改哪些数据，以及测试后如何清理或回查确认最终状态。

复杂链路测试必须在报告里写清角色和 token 类型，但不能写完整 token。

执行历史脚本时必须显式传入用户确认的 `baseUrl` 和 token；不要直接使用脚本内置的默认环境、默认 token 或默认账号。

### 第三步：执行验证

按风险从基础到业务执行：

1. 编译：优先当前模块，其次全项目。
2. 单测：优先本次改动相关 Service、Mapper、工具方法和状态流转。
3. 接口：按角色和接口顺序发起请求，记录请求、响应和断言。
4. 主链路：按业务顺序串联接口，例如创建、查询、变更、回查、注销。
5. 异步链路：WebSocket、MQ、Redis、ES、事务后通知要验证事件是否到达、顺序是否正确。
6. SQL：校验字段、条件、JOIN、分页、逻辑删除、索引命中风险。
7. 回归：检查老接口、老枚举、老状态、空数据和历史数据兼容。
8. 清理和回查：涉及写接口或状态变更时，测试后执行清理动作，或至少回查确认最终状态符合预期。

失败时不要只报错，要记录失败步骤、请求信息、响应信息、推测原因和下一步建议。

应测场景最低要求：

- 主链路 happy path 必须验证。
- 接口契约变更必须验证请求方式、路径、必填参数、出参字段、错误码和兼容性。
- JSON 入参必须验证合法请求体、字段类型、嵌套结构、数组结构、Content-Type 和 DTO 绑定结果。
- 如果接口包含前端禁止传字段或后端自动推导字段，必须验证传入伪造值时以后端契约为准。
- 如果涉及权限、状态、幂等、SQL、Redis/MQ/ES/WebSocket、旧链路或多 agent 合并，必须把对应场景列入清单并验证；不涉及时标 `不适用` 即可。

未验证原因只允许使用清晰分类：

- 缺 token / 缺环境 / 缺测试数据 / 外部依赖不可用 / 需要前端配合 / 需要测试环境权限 / 当前需求不涉及 / 用户要求跳过。

如果未验证原因是文档口径不一致、接口契约不清或 SQL 未确认，应标为 `阻塞`，并退回对应阶段。

### 第四步：生成报告

如果存在需求目录，先写入测试轮次产物：

- `07-test-report.md` 不存在时，从 `../ggg-workflow-shared/assets/workflow/templates/test-report-index-template.md` 创建。
- `test-rounds/` 不存在时创建目录；本轮明细从 `test-report-round-template.md` 创建，命名为 `test-rNN.md`。
- 每次测试新增一个 `T1/T2/T3...` 轮次，不覆盖旧轮次；补测或修复后复测也新增下一轮。
- `07-test-report.md` 只维护轮次索引、当前测试结论和未关闭缺口。
- `test-rNN.md` 写本轮基本信息、测试结论、应测场景清单、执行记录、缺口和阻塞。
- 如果同时生成接口专项报告，把路径写入 `07-test-report.md` 和本轮 `test-rNN.md`。
- 如果没有需求目录但存在 quick 记录，默认只更新 `quick.md` 的验证记录和 Review / 测试结论，不创建 `07-test-report.md` 或 `test-rounds/`。只有用户要求报告、接口验证复杂、或存在未关闭风险时，才按普通报告方式落地，并在最终回复中给出等价的测试轮次摘要。

接口专项报告优先落地到：

- 需求目录：`reports/api-tests/`
- 若是单项目接口测试：目标项目根目录 `reports/api-tests/`

报告命名建议：

```text
<case-name>-YYYYMMDD_HHMMSS.md
```

报告至少包含：

- 测试标题。
- 生成时间。
- 服务地址。
- token 类型和脱敏 token。
- 测试结果概览。
- 应测场景清单和逐项状态。
- 每个步骤的 PASS / FAIL。
- 请求信息。
- 实际发送的 JSON 请求体、Content-Type 和 DTO 绑定口径。
- 响应信息。
- WebSocket / MQ / Redis / ES 事件记录。
- SQL / 配置 / 缓存 / 异步验证证据。
- 测试数据清理或最终状态回查结果。
- 失败原因和后续建议。

### 第五步：输出验证结论

结论只允许三种：

- `通过`：所有关键应测场景已验证通过，非关键未验证项有明确原因且风险可接受。
- `需补测`：当前验证通过一部分，但缺少 token、环境、数据、异步事件、旧链路回归或非阻塞场景。
- `阻塞`：编译失败、接口失败、主链路失败、数据错误或关键异步事件失败。

如果存在需求目录，且本轮测试结论为 `通过`，在写入 `07-test-report.md` 和本轮 `test-rounds/test-rNN.md` 后推进到交付完成：

```bash
py ../ggg-workflow-shared/scripts/workflow_cli.py complete --feature-dir <feature-dir> --test-passed
```

关键场景失败或未验证时必须 `阻塞`：

- 编译失败。
- 主链路失败。
- 本次涉及的接口契约、JSON 入参、权限、SQL、状态流转、幂等、异步消息或旧链路回归无法判断。
- 本次涉及的前端禁止传字段或后端自动推导字段未验证。
- 写接口产生脏数据但无法清理或回查。

最终回复必须包含：

- 验证结论。
- 跑了哪些命令或脚本。
- 应测场景清单里哪些已验证、哪些未验证、哪些阻塞。
- 报告文件路径。
- 失败或缺口。

## 参考样式

已有需求目录示例：

- `ggg/features/20260323-辅学教练工作台/scripts/test_virtual_classroom_flow.py`
- `ggg/features/20260323-辅学教练工作台/reports/api-tests/virtual-classroom-flow-*.md`
- `ggg/features/20260323-辅学教练工作台/06-前后端联调场景说明.md`

该示例体现的规则：

- 脚本参数包含 `--base-url`、`--coach-token`、`--student-token`。
- HTTP 请求使用 `ACCESS-TOKEN`。
- WebSocket 先通过 HTTP 获取 `wsToken` 或 `screenToken`。
- 报告记录请求、响应、WebSocket 事件。
- 报告和控制台输出 token 必须脱敏。

## 硬约束

- 不对线上环境执行写接口，除非用户明确授权并说明回滚方式。
- 不在最终回复和报告中泄露完整 token、Cookie、密码。
- 不伪造测试结果；没跑就是没跑，失败就是失败。
- 不把接口 HTTP 200 等同业务成功，必须检查业务 `code` 或项目实际成功标识。
- 不在 Windows PowerShell 中手拼 curl JSON 请求体；必须使用能保证合法 JSON 的方式发送。
- 不把 JSON 序列化、Content-Type 或 DTO 类型绑定问题误判成业务逻辑失败。
- 不用“覆盖率百分比”替代应测场景清单。
- 不把 worker 局部验证等同 coordinator 合并后的整体验证。
- 不只跑 happy path；本次涉及状态、权限、空数据、重复操作、异步事件时必须覆盖关键边界。
- 不跳过本次涉及的前端禁止传字段、后端自动推导字段、权限隔离、状态流转、幂等和旧链路回归等关键场景。
- 涉及写接口、状态变更、MQ、Redis、ES 或道具发放时，必须说明测试数据影响，并在测试后清理或回查确认；无法清理时报告里必须标记残留数据和影响范围。
- 测试失败后不默认改代码，除非用户明确要求修复。

## 完成标准

- 已执行和本次改动风险匹配的编译、单测、接口或主链路验证。
- 已建立应测场景清单，并对每个场景标记 `已验证 / 未验证 / 阻塞 / 不适用`。
- 所有关键应测场景都有证据或明确阻塞原因。
- full 模式或用户要求报告时，已生成 Markdown 测试报告，且 token 已脱敏。
- 如果存在需求目录，已更新 `07-test-report.md`，并新增本轮 `test-rounds/test-rNN.md`；多轮测试没有覆盖历史报告。
- 如果是 quick 模式且没有需求目录，已更新 `quick.md` 的验证记录和测试结论；无法落文件时，已在最终回复或普通 Markdown 报告中给出应测场景、执行证据、缺口和结论。
- 已明确结论：`通过`、`需补测` 或 `阻塞`。
- 用户可以直接把报告交给测试、前端或后端继续处理。
