任务初始状态：

| 状态 | 含义 |
|---|---|
| ready | 依赖已满足，可以开始 |
| pending | 等前置任务完成后开始 |
| blocked | 依赖外部确认、SQL 执行、环境或跨项目交付 |

并行组参考：

| 并行组 | 含义 |
|---|---|
| P0 | SQL / 配置 / 环境前置 |
| P1 | 实体、Mapper、DTO、枚举、配置类等基础改动 |
| P2 | 核心 Service / Manager / Facade 编排 |
| P3 | Controller / Dubbo / HTTP 接口与接口契约对齐 |
| P4 | 测试、回归、发布、回滚 |

并行等级说明：

| 等级 | 使用场景 | 必填输出 |
|---|---|---|
| L0 | 不适合并行，或任务太小不值得并行 | 不并行原因、推荐串行顺序 |
| L1 | 只做并行可行性评估，实际仍默认单 agent / 串行 | 并行安全评估矩阵 |
| L2 | 明确准备多 agent 并行实现 | 并行安全评估矩阵、多 agent worker 分配表 |

并行安全评估矩阵（L1 / L2 必填，L0 可删除或写“不适用”）：

| 任务 | 并行组 | 预计写入文件/目录 | 共享依赖 | 契约/表/配置影响 | 冲突结论 | 执行方式 |
|---|---|---|---|---|---|---|
| T | P |  |  | HTTP/Dubbo/SQL/Redis/MQ/ES/配置/无 | 可并行/需串行/先锁契约 | 并行/串行/准备并行 |

> 说明：并行组只是阶段和派工建议，不等于自动并行执行。同一个 Service 主链路、Mapper XML、配置文件、接口契约、Provider/Consumer 契约两端冲突时必须标为 `需串行` 或 `先锁契约`。

多 agent 并行编码分配表（仅 L2 必填；L0 / L1 不要伪造 worker 表）：

| Worker | 任务 | 允许修改文件/目录 | 禁止修改文件/目录 | 只读参考文件/目录 | 依赖任务 | 合并顺序 | 验证命令 |
|---|---|---|---|---|---|---|---|
| Wn | Tn |  |  |  | - | 1 |  |

> 说明：`允许修改文件/目录` 优先写精确文件路径；目录级权限只用于新增文件且要限定唯一子目录或命名前缀。同一个文件不得分配给两个 worker；如果必须改同一个文件，放到同一个 worker 或改为串行。

## 8. 完成定义

- 所有 `ready/pending` 任务均有明确依赖和完成标准
- 每个任务均有 `来源依据`，可追溯到 `Dxx/Cxx/interface/schema`
- `blocked` 任务已写清阻塞原因和解除条件
- SQL / 回填脚本已准备或明确不涉及
- 每个接口明细都有实现任务和测试任务
- 每个关键风险都有防护任务或回归任务
- 关键接口 / 契约验证通过
- 发布和回滚任务已覆盖

## 9. 后续开发注意事项

- 如果后续编码实现偏离 `02-design.md` 或 `03-tasks.md`，先回写对应权威文档
- 如果表结构、SQL、配置、接口有临时调整，优先同步 `02-design.md`、`04-schema.sql`、`03-tasks.md`
- 如果发布方式、回退动作与原计划不一致，优先回写 `02-design.md` 或 `03-tasks.md`
- 遗留问题需要后续跟进时，直接补到任务或设计文档，不额外新增其他回填文档
